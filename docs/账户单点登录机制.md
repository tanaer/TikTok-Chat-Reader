# 账户单点登录机制

## 功能概述

系统新增全局开关 `single_session_login_enabled`，用于控制账号是否只允许保留一个有效登录会话。

- 关闭时：同一账号可在多个浏览器或设备同时登录。
- 开启时：同一账号再次登录会立即使旧登录失效。
- 后台位置：管理后台 `系统设置 -> 登录与安全`。

## 实现方式

本次实现采用“会话版本号”方案，而不是只依赖刷新令牌撤销。

### 1. 用户会话版本

在 `users` 表新增 `session_version` 字段，默认值为 `0`。

- 每次普通登录时，如果单点登录开关开启，则把该用户的 `session_version` 加 `1`。
- 新签发的访问令牌会把当前 `session_version` 写入 JWT 的 `sessionVersion` 字段。
- 服务端在鉴权中间件中会同时校验：
  - JWT 是否合法
  - 用户状态是否为 `active`
  - JWT 中的 `sessionVersion` 是否等于数据库中的 `users.session_version`

只要用户在别处重新登录，旧令牌中的版本号就会落后，从而立即失效。

### 2. 刷新令牌绑定会话版本

在 `refresh_tokens` 表新增 `session_version` 字段。

- 登录或刷新令牌轮换时，刷新令牌会绑定当前会话版本。
- `/api/auth/refresh` 在签发新令牌前，会校验刷新令牌版本与 `users.session_version` 是否一致。
- 如果版本不一致，会返回 `401` 与 `SESSION_REVOKED`，避免旧设备依赖 refresh token 恢复登录。

### 3. 登录时踢掉旧会话

当单点登录开关开启时，登录接口会在事务中执行以下动作：

1. 锁定当前用户行。
2. 撤销该用户历史 refresh token。
3. 递增 `users.session_version`。
4. 签发新的 access token / refresh token。

这样可以保证同一时刻只有最新一次登录生成的会话有效。

## 相关接口行为

### 被顶下线

当旧登录继续访问受保护接口时，服务端返回：

- HTTP 状态码：`401`
- 错误码：`SESSION_REVOKED`
- 错误信息：`账号已在其他地方登录，请重新登录`

前端收到该错误后，会清理本地令牌并跳转到登录页。

### 刷新令牌轮换并发保护

前端访问受保护接口时，如果 `access token` 过期，会自动调用 `/api/auth/refresh` 使用 `refresh token` 换发新令牌。

由于系统启用了“refresh token 轮换”：

- 每次刷新成功后，旧 refresh token 会立即作废
- 同时签发新的 access token / refresh token

因此前端必须避免同一时刻并发发起多次刷新请求。否则会出现：

1. 多个接口同时因 `401` 进入刷新流程
2. 第一条刷新请求成功，旧 refresh token 被作废
3. 后续并发刷新仍携带旧 refresh token，请求失败
4. 页面误以为登录态彻底失效，刷新页面后被跳回登录页

当前前端已补上刷新去重保护：

- 同一时刻只允许一个 `/api/auth/refresh` 请求在进行
- 其余请求复用同一个刷新 Promise 等待结果
- 如果某个较晚返回的刷新请求发现本地 refresh token 已被更换，不再把会话误判为失效
- 多窗口 / 多标签页之间也会通过本地 refresh 锁避免重复轮换同一个 refresh token
- 如果刷新接口只是临时失败（如服务重启、数据库瞬时异常），前端不会立刻清空本地登录态并跳转登录页

这样可以避免“用户只在一个地方登录，但过一段时间刷新页面后掉线”的假性登出问题。

### 修改密码 / 重置密码

为了立即使旧登录失效，以下操作也会递增 `session_version` 并撤销 refresh token：

- `PUT /api/auth/change-password`
- `POST /api/auth/reset-password`

## 安全说明

本次改动遵循最小暴露原则：

- 未新增任何用户侧敏感响应字段。
- `single_session_login_enabled` 仅在管理员设置接口中可见，已加入公共配置过滤列表。
- 未向普通用户接口暴露 `session_version`、refresh token 绑定关系或其他内部鉴权细节。

## 后续建议

如后续需要扩展到“允许最多 N 台设备同时登录”，可以在当前方案上进一步演进为独立会话表，并按设备维度管理会话状态。
