# AI点数计费口径

日期：2026-03-08

## 目的

统一系统内 AI 能力的用户侧计费单位，并给新能力落地时提供一致的定价判断标准。

从本次调整开始，前台、用户中心、管理台中的 AI 用量展示，统一使用“点”作为计费单位，而不再使用“次”。

## 当前口径

- 用户侧展示：`AI 点数`
- 套餐赠送：`AI 额度 X 点/月`
- 充值商品：`AI 点数包`
- 余额展示：`剩余 X 点`
- 使用展示：`已用 X 点`

## 说明

- 当前数据库字段与内部实现仍沿用 `ai_credits_*` 命名。
- AI 点数包价格统一按“元”配置、展示和余额扣款。
- `ai_credit_packages.price_cents` 作为历史兼容存储字段保留，但接口与前端统一使用 `priceYuan`。
- 后续新增 AI 能力时，应默认采用“按点消耗”的口径，而不是“按次消耗”的口径。

## CNY 估算基线

参考现有点数包测算示例：

- `10 点 / ¥10` → `¥1.00/点`
- `100 点 / ¥50` → `¥0.50/点`
- `500 点 / ¥200` → `¥0.40/点`

因此真实成交视角下，主流点价可按 `¥0.4 ~ ¥0.5/点` 估算。

## 已落地能力映射

- AI 用户分析（旧版文本分析）：默认消耗 `1 点`
- AI 客户分析（上下文版）：默认消耗 `3 点`
- AI 直播复盘（单场版）：默认消耗 `20 点`
- AI 直播复盘需同时满足“套餐已开通权益 + 账户点数充足”两个条件
- 已生成内容再次查看：不重复扣点

## 扣点确认规则

- 所有会真实消耗 AI 点数的用户操作，默认必须二次确认后才能执行。
- 确认文案必须明确展示“本次会扣点”，并写清楚单次或本批次预计消耗。
- 服务端对查看缓存结果、命中复用任务、预检查失败这类 `0 点` 路径不强制确认；前端若暂时无法提前判定，也应按“可能扣点”给出明确提示。
- 批量类操作应优先展示“最多消耗 X 点”的上限提示，避免用户无感知连续扣点。

## AI客户分析定价方案

### 当前上线口径

- 命中个人缓存再次查看：`0 点`
- 首次生成：`3 点 / 次`
- 强制重新分析：`3 点 / 次`
- 管理员发起：`0 点`

### 为什么定为 3 点

- 已从“只看弹幕文本”升级为“结构化上下文 + 模型标签 + 策略建议 + 话术输出”。
- 按 `¥0.4 ~ ¥0.5/点` 估算，`3 点` 约合 `¥1.2 ~ ¥1.5`，具备明确感知价值。
- 相比旧版 `1 点` 的轻分析，`3 点` 更符合营收目标，也能拉开和普通文本分析的价格带。
- 同时仍显著低于 `20 点 / 场` 的 AI直播复盘，不会打乱现有 AI 产品梯度。

### 后续偏贵档建议

如果后续继续扩展能力，可采用更高档位：

- 单客户分析增强版（增加召回动作树 / 维护脚本）：`5 点 / 次`
- 批量 Top 客户分析：`20 ~ 30 点 / 批`
- 高价值客户召回方案包：`30 ~ 50 点 / 批`

建议原则：

- 只要新增了更多系统计算、更多上下文字段、更多动作建议模板，就应同步提高点数门槛。
- 新功能优先采用“价值定价”而不是“成本定价”；在用户能直接拿去运营成交的场景里，宁可略贵，不宜过轻。

## 结构化数据源与模块化分析定价补充

### 后台结构化数据源测试

- 管理后台的 `结构化数据源` 菜单仅用于系统事实校验与 Prompt 调试。
- 该类测试默认 `0 点`，不计入用户余额。
- 原因是它不产生用户侧可交付的 AI 内容，本质属于后台配置与验算能力。

### 模块化 AI 增强版建议定价

如果后续把“结构化数据源 + Prompt 编排”继续做深，建议直接拉开价格带：

- AI客户分析增强版（更多数据源、更多证据分区、更多动作脚本）：`5 点 / 次`
- AI直播复盘增强版（复盘主输入 + 经营基线 + 新增关注 + 更多结构化信号）：`25 点 / 场`
- 批量客户分析（Top 客户 / 潜力客户批量）：`20 ~ 30 点 / 批`

### 定价判断原则

- 只要用户拿到结果后可以直接用于老板汇报、主播承接、场控转化、客户维护，就不应按“普通文本分析”定价。
- 模块化后的 AI 不是“多喂一点字”，而是“系统先算好事实，再生成可执行策略”，价值显著高于轻分析。
- 在营收优先前提下，建议默认采用偏贵一档的定价，再根据实际转化率回调，而不是先低价再补涨。

## 后续建议

- 将套餐中的 `ai_credits_monthly` 解释为“每月赠送 AI 点数”。
- 后续如需支持更精细账务，可区分“月赠点数”和“购买点数”两类余额池。
- 若未来上线批量客户分析，建议在前台明确展示“单次扣点 / 批量扣点 / 重跑扣点”三种规则，避免用户无感知消耗。
