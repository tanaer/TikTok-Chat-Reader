# 套餐与扩容包定价及上架规则

## 注册赠送套餐的上架规则

`gift` 套餐用于承接“注册赠送”订阅记录，本质上是一个后台维护的系统套餐，不应该因为有新用户注册就被自动上架给所有用户购买。

当前规则调整为：

- 注册时如果系统开启了赠送房间数和赠送天数，仍会自动创建或更新 `gift` 套餐，并给新用户发放对应赠送订阅。
- 自动创建 `gift` 套餐时默认写成 `is_active = false`。
- 如果 `gift` 套餐已存在，注册流程只同步名称、房间额度、每日建房额度和价格字段，不再覆盖管理员手动设置的上架状态。
- 因此，管理员手动下架后，后续新增注册用户不会再把该套餐重新上架。

这样既保留了注册赠送能力，也避免“系统赠送套餐混进公开售卖列表”。

## 套餐价格删除规则

后台套餐编辑支持对月价、季价、年价分别做删除操作，删除的实际含义是：

- 将对应价格置为 `0`
- 前台不再展示该计费周期
- 用户也不能再购买该计费周期

为了避免误操作，删除价格时需要二次确认，并明确提示“删除后该周期会下线，用户无法购买”。

同时增加一条保护规则：

- 一个套餐至少要保留一个大于 `0` 的价格周期
- 不允许把月价、季价、年价全部删空

服务端和后台前端都会共同校验，防止出现“套餐还在，但所有周期都不可买”的脏状态。

## 前台套餐周期展示规则

会员套餐前台统一按“价格大于 0 才显示该周期”的原则处理：

- 月价 `> 0` 才显示月付
- 季价 `> 0` 才显示季付
- 年价 `> 0` 才显示年付

如果不同套餐可购买周期不一致：

- 套餐对比页优先显示每个套餐自身第一个可购周期
- 只有当所有套餐都同时支持某个周期时，顶部公共周期切换器才显示该周期

这样可以避免某个周期被删除后，前台仍把它当成可买周期展示。

## 房间扩容包业务逻辑

房间扩容包属于主套餐之外的增购项，但不再独立成一段新的订阅周期，而是跟随当前主套餐有效期：

- 只有当前存在有效企业版会员时，才允许购买扩容包。
- 购买成功后，会向 `user_room_addons` 写入一条生效记录，开始时间为购买时间，结束时间直接对齐当前主套餐到期时间。
- 扩容包只覆盖“当前这一段会员有效期”，不会在用户后续续费时自动顺延。
- 价格按当前会员周期对应的扩容包基准价，再按“剩余天数 / 当前会员周期总天数”做折算。
- 配额计算时，系统会统计当前用户所有“已开始、未过期且 active 的扩容包”，把它们的 `room_count` 累加到总房间上限。
- 如果主套餐本身是无限房间，则扩容包不会再额外叠加房间上限，因为总额度已经是无限。
- 扩容包支持配置 `单账户限购次数`：达到次数上限后，该账号不能再重复购买同一扩容包。
- `单账户限购次数` 留空或设为 `0` 视为不限购。

这样可以避免主套餐和扩容包各自维护独立到期时间，也避免出现“会员过期但扩容还在生效”的理解成本。

## 单账户限购次数规则

- 配置位置：后台 `扩容包配置`
- 存储字段：`room_addon_packages.per_user_purchase_limit`
- 口径：按账号维度统计该扩容包的历史购买记录总次数，达到上限后不可继续购买
- 生效方式：
  - 前台扩容包列表展示限购次数与当前已购次数
  - 不满足条件时前台按钮置灰
  - 后端 `purchase-addon` 接口再次强校验，避免绕过前端直接下单
- 默认建议：
  - 标准扩容包可先设置为 `1~3` 次
  - 若需要长期开放重复叠加，则留空不限购

## 价格评估结论

默认扩容包基准价如下：

- `5房间扩容包`：月 `49` / 季 `129` / 年 `469`
- `10房间扩容包`：月 `89` / 季 `239` / 年 `849`
- `20房间扩容包`：月 `159` / 季 `429` / 年 `1519`

这些价格不再表示“用户买下一个独立月包/季包/年包”，而是作为折算基准价使用：

- 如果当前企业版是月付，就按月价折算本期剩余天数
- 如果当前企业版是季付，就按季价折算本期剩余天数
- 如果当前企业版是年付，就按年价折算本期剩余天数

按“单房月成本”看：

- 5 房包：约 `9.8 元/房/月`
- 10 房包：约 `8.9 元/房/月`
- 20 房包：约 `7.95 元/房/月`

对比主套餐月付单房成本：

- 基础版：`99 / 5 = 19.8 元/房/月`
- 专业版：`299 / 20 = 14.95 元/房/月`
- 企业版：`999 / 100 = 9.99 元/房/月`

可以得到两个判断：

- 扩容包存在明显的阶梯优惠，买得越多越便宜，方向合理。
- 扩容包整体并不贵，甚至比基础版、专业版的单房成本更低，接近或略低于企业版单房成本。

因此，如果产品定位是“鼓励老用户在原套餐上灵活增购”，当前价格偏友好；如果希望更多用户直接升级主套餐而不是长期堆扩容包，可以考虑未来适度提高小包价格，或限制低阶套餐叠加太多扩容包。
