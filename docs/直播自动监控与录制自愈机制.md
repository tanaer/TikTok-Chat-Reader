# 直播自动监控与录制自愈机制

## 背景

直播房间存在两套容易被混淆的标识：

- `uniqueId / room_id`：系统房间列表、房间详情和用户配置使用的业务房间标识。
- `numeric_room_id`：TikTok WebSocket 建连、`check_alive` 判活和录制链路使用的真实直播间号。

当 TikTok 公开页面解析波动、缓存房间号失效、或数据库里房间开关字段存在历史 `NULL` 脏数据时，容易出现以下现象：

- 房间明明在播，但列表不显示“直播中”
- 自动监控没有拉起
- 手动进入房间详情点击“连接”后，很快拿到真实 `roomId`
- 视频自动录制或实时数据采集表现出“手动能连，自动不行”的分叉

## 修复目标

本次机制调整的目标有三点：

1. 自动监控链路与手动连接链路使用同等级的缓存房间号回退与重试策略。
2. 房间配置字段统一按 camelCase / snake_case 双兼容读取，避免历史数据或 PostgreSQL 字段映射差异导致监控、录制开关失效。
3. 数据库启动时自动修正历史 `NULL` 房间开关，避免代码修复后仍被旧数据拖回异常状态。

## 自动监控建连机制

`auto_recorder.js` 在后台轮询需要监控的房间时，仍然以业务侧 `roomId` 为主索引，但建连时会优先读取 `room.numeric_room_id`：

1. 每轮自动巡检开始前，会先刷新一次运行中的 Euler Key 池与代理配置，再复用该轮配置执行整批预探活与建连。
2. 如果数据库已有 `numeric_room_id`，不会立刻盲连，而是先对该值执行 `check_alive`；确认仍在播时再直接复用。
3. 若缓存房间号已经过期，则改走 live 页快照；如果页面被 TikTok WAF 挡住，再退到 `fetchRoomInfoFromApiLive()` 获取当前场次的房间号。
4. 一旦预探活发现当前场次的 `numeric_room_id` 已变化，会先写回 `room.numeric_room_id`，然后才进入真正的 WebSocket 建连。
5. 若缓存房间号建连失败，并且错误属于可重试类型，例如超时、TikTok 200 异常响应、WebSocket 建连失败等，会自动清空旧缓存并退回到“重新解析房间号”。
6. 自动监控新增与手动连接一致的最多 3 次重试，避免一次解析抖动就把房间长期留在“未直播”。
7. 连接建立后，优先使用 `webcast/room/check_alive` 对 `numeric_room_id` 做判活，再决定是否把房间标记为“直播中”。

这样做的结果是：

- 手动连接拿到过的真实房间号，会反哺自动监控
- 自动监控不再明显弱于手动连接
- TikTok HTML/API 临时抖动时，已知房间号的直播间不会轻易掉监控

## 僵尸连接回收机制

之前有一类高频故障：

- 进程里某房间仍残留 `activeConnections`
- WebSocket 表面上还是 `connected`
- 但实际上长期没有事件、也没有通过 `check_alive` 确认 live
- 自动巡检因为看到“已经连着”就跳过该房间
- 房间列表又因为这条占坑连接而压住 Redis/事件兜底

结果就是：

- 房间真实在播
- 列表却显示“未开播”
- 自动监控也不再重新拉起
- 只有人工点一次“连接”后才可能恢复

修复后：

1. 自动巡检在每轮扫描时，会先判断已有连接是否真正“可用”。
2. 可用连接的标准不再只是 `wsConnected=true`，而是满足以下任一条件：
   - 已通过 `check_alive` / 实时事件确认 live
   - 仍处于预探活刚确认 live 的短 TTL 窗口
3. 对于只是“挂着 socket、但没有 live 证据”的连接，会先回收，再重新纳入自动建连。
4. 手动连接入口也采用同样逻辑，不再把这种僵尸连接误判成“已连接”。

## 自动录制触发机制

自动录制与实时数据采集是两条相关但独立的链路：

- 实时数据采集：依赖 AutoRecorder 的 WebSocket 建连与事件落库
- 视频自动录制：依赖房间配置中的 `is_recording_enabled` 与 `recording_account_id`

本次修复前，部分代码路径只读取 snake_case 字段，例如 `is_recording_enabled`，但数据库查询结果经 `db.js` 转换后返回 camelCase 字段，例如 `isRecordingEnabled`。这会导致：

- 房间明明已配置自动录制，运行时却判断成未开启
- 或历史 `NULL` 值被继续写回数据库，导致配置失真

修复后：

1. `manager.updateRoom()` 统一兼容 `isMonitorEnabled / is_monitor_enabled`。
2. `manager.updateRoom()` 统一兼容 `isRecordingEnabled / is_recording_enabled`。
3. `manager.updateRoom()` 统一兼容 `recordingAccountId / recording_account_id`。
4. `auto_recorder.js` 的自动录制触发统一走 `ensureAutoRecordingForRoom()`，并兼容上述两套字段命名。

## 手动连接与自动监控统一

之前“手动点连接后马上恢复”的一个核心原因是：

- 手动连接链路在某些情况下会重新拿到新房间号
- 自动巡检却仍被旧 `numeric_room_id` 或占坑连接卡住

修复后，手动连接入口与自动巡检统一为同一套前置判断：

1. 先读缓存 `numeric_room_id`
2. 先做预探活，确认是否真实在播
3. 必要时刷新到新的 `numeric_room_id`
4. 若房间当前已有不可用连接，先回收
5. 再发起真正的 WebSocket 连接

这样可以避免“手动能连、自动不行”继续长期分叉。

## 当前效率评估

当前自动监控的稳定性瓶颈，不再主要是“有没有多把 Key”，而是下面三件事共同决定：

1. TikTok live 页和公开接口是否被 WAF 干扰。
2. 运行中的 Key 池是否真的被进程及时刷新并参与轮换。
3. 自动巡检是否把不可用连接及时回收，让房间重新进入探活。

修复后，现阶段的调度特征调整为：

- 监控扫描周期默认 `5 分钟`
- Web 服务启动完成数据库初始化后，会先装载 `euler_api_keys` 表中的启用 Key
- 每轮自动巡检开始前，会同步刷新一次运行时 Key 池与代理配置
- 每轮预探活并发上限改为“按活跃 Key 数动态放大，再受安全上限约束”
- 真实连接启动之间固定至少间隔 `1.5 秒`
- 遇到 `Too many connections started` 会进入全局冷却 `2 分钟`
- 遗留事件清理与场次合并只在确认房间确实在线、准备建连后才执行

这意味着即使后台录了 10 个 Key，也不等于系统会并行拉 10 路真实 WebSocket 连接。当前代码仍然是“预探活放宽、真实建连保守”的策略：

- Key 的主要价值是提高轮换、容错和预探活吞吐，不是线性放大真实建连并发
- 真实建连依然会被启动间隔和全局冷却约束
- 当监控房间数量远大于正在直播的房间数量时，新的调度方式能显著减少“对离线房间做无意义维护开销”

所以 10 个 Key 会提升稳定性上限，但不会直接让 259 个启用房间在 5 分钟一轮里都能被秒级覆盖。当前修复已经把“只装了 5 个环境变量 Key”和“每个房间预检前都跑一次重型清理”这两类系统性浪费去掉，但如果还要继续提速，下一步应该考虑的是：

1. 把扫描周期从全量轮询改成“热点房间优先 + 冷房间退避”的分层调度。
2. 让预探活并发同时参考代理能力、最近限流状态和房间优先级，而不只看 Key 数量。
3. 给 Key 池运行态增加可观测性，让后台能直接看到 10 个 Key 当前到底有几把在参与轮换、几把在冷却。

## 直播时间窗优先级

考虑到很多直播间的开播时间长期相对固定，自动监控现在会基于历史场次自动推导“高概率开播时间窗”，而不是永远把所有房间当成同一优先级：

1. 从近 `21` 天历史 `session` 中读取每个房间最近最多 `12` 个样本场次。
2. 把每个历史场次映射到北京时间小时段，统计哪些小时最常出现。
3. 如果某房间存在明显集中的高频小时段，则生成该房间的“优先扫描时间窗”。
4. 当前时间落在时间窗附近时，这个房间会被提升到更高扫描优先级。
5. 远离时间窗、且近期没有其他活跃信号的房间，会自动降低扫描频率。
6. 没有足够历史样本的房间，不会因为画像缺失而被强制降级，仍按普通路径继续扫描。

这层机制解决的不是“解析不到房间号”本身，而是把有限的探活预算优先分配给“当前时段最可能开播”的房间，从而减少大房间池下的无效轮询。

## 后台自动监控诊断

为了让“明明在播但没监测到”不再只能靠人工猜测，后台“场次运维”页新增了自动监控诊断面板。它不会暴露原始 Key、Cookie 或上游密钥，只展示运行态摘要和掩码后的样本：

1. 扫描摘要：
   - 启用监控房间数
   - 当前可立即尝试的房间数
   - 时段退避跳过数
   - 正在连接、已连接、已预确认直播的数量
2. Key 池摘要：
   - 总 Key 数、活跃 Key 数、冷却中的 Key 数
   - 最近连接路径是 Euler 直连还是 HTML / API 回退
   - 最近房间解析、`check_alive` 请求量
3. 房间级诊断：
   - 当前高优先级房间
   - 被历史时段退避的房间
   - 连接冷却中的房间
   - 连续失败中的房间
4. 近期失败记录：
   - live 页快照失败
   - 房间号解析失败
   - 建连启动限流

这样做的意义是把“效率不够”拆成几个可操作的问题：

- 如果是 Key 池全冷却，优先解决 Key 可用性。
- 如果是全局建连冷却频繁触发，说明当前瓶颈是连接启动节奏，而不是单纯 Key 数量。
- 如果是大量房间长期处于 `snapshot_unavailable` 或 `room_id_resolution` 失败，说明更偏向 TikTok 上游页面/API 可达性问题。
- 如果大量房间都落在冷时段退避里，则应优先校准房间画像或补充历史样本，而不是盲目提高全量扫描频率。

## 受控直连兜底

还有一类之前容易漏掉的房间：

- 直播页快照拿不到
- API 也暂时不给有效结果
- 自动巡检在预探活阶段直接判成 `snapshot_unavailable`
- 但底层连接器如果继续往下走，仍有机会通过已有缓存房间号、签名链路或后续连接流程连上

这会形成一种典型分叉：

- 自动监控因为“太保守”提前跳过
- 人工点连接或更底层链路却偶尔能恢复

现在自动巡检新增了“受控直连兜底”：

1. 只对高优先级房间开放：
   - 落在历史热点时段附近
   - 或后台手工优先级较高
   - 或综合分数达到阈值
2. 每轮只放出很小的兜底预算，不会对所有 `snapshot_unavailable` 房间都盲连。
3. 对“已有缓存 `numeric_room_id`，但 live 页 / API 探活被 `403`、`429` 或 `user_not_found` 挡住”的房间，还会保留一组更小的独立预算，避免它们被普通热点房间抢光全部兜底名额。
4. 同一个房间还有独立冷却，避免一个抖动房间每轮都反复消耗兜底预算。
5. 如果兜底直连仍失败，仍会回到房间级冷却和失败记录里，不会无限重试。

这层机制的目的不是“暴力建连”，而是补掉“预探活过早放弃”造成的漏监控，同时尽量不重新触发大规模 `Too many connections started`。

## 动态画像与自动衰减

用户担心的一个点是对的: 这类修复不能做成永久白名单，也不能把“某房间以前被拦过”固化成长期规则。正确做法是只持久化最近证据，并且让它自动衰减。

现在自动监控新增了一层短期动态画像，专门覆盖这种房型:

- live 页快照或 API 预探活经常返回 `403`、`429`、`user_not_found`、`snapshot_unavailable`
- 但房间本身已有缓存 `numeric_room_id`
- 更底层直连近期又确实成功过

这层画像的行为约束如下:

1. 画像只在“预探活被拦，但兜底直连真实成功”后建立或增强。
2. 画像会记录最近成功、最近失败、探活错误类型、成功/失败计数和过期时间。
3. 画像默认只保留短 TTL，后续如果继续成功会延长一点，但仍有上限，不会无限续命。
4. 如果后续直连失败、探活恢复正常、或房间确认下播，画像分值会自动下降。
5. 一旦分值被打空、连续失败过多、或超时没有新证据，这个画像会自动删除。
6. 调度时它只会提供“短期加分”和“允许更积极地保留缓存 roomId 直连”，不会跳过预算、冷却和限流保护。

因此这层机制是:

- 持久化最近证据
- 不是固化永久规则
- 重启后仍能保留短期恢复能力
- 但会随着时间和失败自动退场

这样可以解决一类最烦的房间:

- 人工点连接经常能恢复
- 自动巡检却因为 probe 太保守反复放弃

同时又不会把误判房间永久抬到高优先级，避免后面越修越脏。

## 如何判断是否要继续加 Key

现在“要不要加 Key”不能只看房间总数，要结合后台诊断一起判断：

1. 适合加 Key 的情况：
   - 活跃 Key 数经常明显小于总 Key 数
   - `rateLimitCount`、`disabledCount` 持续增长
   - 预探活并发被 Key 数卡住，而热点房间仍很多
2. 不适合只靠加 Key 的情况：
   - 后台频繁出现全局 `Too many connections started`
   - 近期失败主要是 `snapshot_unavailable`
   - 房间列表里大量房间本来就处于冷时段，被时段退避主动降频
3. 经验判断：
   - Key 的价值主要体现在提升预探活吞吐、轮换和容错
   - 它不能线性提升 WebSocket 建连速度
   - 当问题出在 TikTok WAF、HTML 解析、API 抖动或建连启动速率限制时，单纯继续加 Key 收益会迅速下降

## 数据库自愈策略

为了兼容历史数据，本次在 `db.js` 启动初始化里增加了轻量修复：

- `room.is_monitor_enabled IS NULL` 时自动回填为 `1`
- `room.is_recording_enabled IS NULL` 时自动回填为 `0`

这两条更新被放在启动引导版本短路之前执行，因此即使数据库已经完成过大版本 bootstrap，后续服务重启时仍能自动修正历史脏数据。

## 当前边界

本机制修复的是“系统没有自动识别直播、没有自动拉起采集/录制”的问题，不会擅自替用户修改业务意图：

- 如果房间 `is_recording_enabled = 0`，系统仍不会自动启动视频录制
- 但实时直播识别、事件采集与列表直播状态应恢复正常

如果用户希望该房间自动录制视频，还需要在房间配置中显式开启自动录制并配置可用录制账号。
