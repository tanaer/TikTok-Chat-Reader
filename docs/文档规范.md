# 文档规范

## 目的

`docs/` 目录用于保存系统设计、架构说明、关键机制、性能方案和重要业务流程文档。

后台“系统文档”页面会直接读取这里的 Markdown 文件。

## 编写要求

### 1. 必须使用 Markdown

所有系统设计文档都应使用 `.md` 文件保存。

### 2. 第一个一级标题必须清晰

后台会读取文档中的第一个：

```md
# 标题
```

并把它作为后台文档列表的显示名称。

因此标题必须直接说明文档内容，避免写成“说明”“记录”“文档”这类过于模糊的名字。

### 3. 内容要可维护

建议每篇文档至少包含：

- 背景
- 目标
- 方案
- 影响范围
- 后续建议

### 4. 什么时候必须更新文档

以下变更完成后，应同步更新 `docs/`：

- 系统架构调整
- 关键数据流调整
- 权限模型调整
- 配额、支付、分析、录制等核心机制变更
- 重要性能优化
- 后台管理能力新增或重大变更

## 建议命名

优先使用清晰的中文文件名，例如：

- `系统设计总览.md`
- `房间列表性能优化机制.md`
- `支付链路说明.md`
