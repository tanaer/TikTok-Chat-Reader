# 场次运维与自动归档配置中心

## 背景

直播监控链路里，场次归档、断线续场、陈旧 LIVE 清理、碎片场次合并原本分散在多处逻辑里：

- `AutoRecorder` 负责断线后的延迟归档与续场判断
- `manager.js` 负责陈旧事件归档、孤儿事件修复、空场次删除、缺失场次重建
- `server.js` 暴露了若干历史维护接口

这会带来三个问题：

1. 参数分散，阈值改动需要改代码或记住多个默认值
2. 运维行为不可视，难以回答“刚才到底执行了什么、成功了没有”
3. 手动修复入口和自动任务入口彼此割裂，不利于线上排障

因此本次改造把它们统一收敛为一套“场次运维配置中心”。

## 设计目标

- 把关键阈值提取到 `settings` 表，支持后台可配置
- 自动任务与手动任务统一复用同一套执行服务
- 所有任务写入运维日志，支持后台可视化查看
- 保留旧接口，但统一要求管理员权限并转发到新服务
- 后台提供独立菜单，集中展示概览、配置、手动执行、最近日志

## 配置项

当前场次运维中心统一管理以下变量：

- 断线续场窗口：`session_maintenance_resume_window_minutes`
- 断线延迟归档：`session_maintenance_archive_delay_minutes`
- 启动时陈旧 LIVE 清理开关：`session_maintenance_startup_cleanup_enabled`
- 陈旧 LIVE 清理间隔：`session_maintenance_stale_cleanup_interval_minutes`
- 大间隔切场阈值：`session_maintenance_stale_gap_threshold_minutes`
- 历史年龄强制拆分阈值：`session_maintenance_stale_split_age_minutes`
- 整段陈旧事件归档阈值：`session_maintenance_stale_archive_all_age_minutes`
- 启动时碎片场次扫描开关：`session_maintenance_startup_consolidation_enabled`
- 碎片场次巡检间隔：`session_maintenance_consolidation_interval_minutes`
- 最近碎片场次扫描窗口：`session_maintenance_consolidation_lookback_hours`
- 自动合并间隔阈值：`session_maintenance_consolidation_gap_minutes`
- 手动合并间隔阈值：`session_maintenance_manual_merge_gap_minutes`
- 日志保留天数：`session_maintenance_log_retention_days`

## 核心实现

### 1. 统一配置服务

`services/sessionMaintenanceService.js` 提供统一能力：

- 读取并标准化配置
- 兼容 `snake_case` 存储键与后台 `camelCase` 提交字段
- 统一执行维护任务
- 统一写入日志
- 提供概览和日志查询接口

### 2. 自动任务调度

`AutoRecorder` 在启动时会读取场次运维配置，并负责：

- 启动补偿任务
  - 陈旧 LIVE 清理
  - 最近碎片场次扫描
- 定时任务调度
  - 按配置间隔执行清理与扫描
- 断线归档延迟调度
  - 按配置窗口保留续场机会
  - 配置更新后重排待归档定时器

### 3. 运维日志

系统新增 `session_maintenance_log` 表，用于记录：

- 任务类型
- 触发来源
- 房间号（如有）
- 状态（运行中、成功、失败、已计划、跳过、已取消）
- 配置快照
- 执行摘要
- 错误信息
- 开始时间、结束时间、耗时

日志既记录“维护任务”，也记录“生命周期事件”，例如：

- 断线后已创建延迟归档
- 在续场窗口内取消归档并恢复场次
- 开播前发现历史遗留事件并先行清理
- 实际归档成功或失败

## 后台接口

### 管理后台专用接口

全部位于 `/api/admin/session-maintenance/*`，并要求管理员身份：

- `GET /api/admin/session-maintenance/config`
  - 返回当前配置、默认值、字段定义
- `PUT /api/admin/session-maintenance/config`
  - 保存配置，并触发 `AutoRecorder` 热刷新调度
- `GET /api/admin/session-maintenance/overview`
  - 返回配置、统计、任务概览、最新日志、运行时快照
- `GET /api/admin/session-maintenance/logs`
  - 返回最近日志，支持 `limit/taskKey/status/roomId` 过滤
- `POST /api/admin/session-maintenance/actions/:action`
  - 手动执行维护动作

### 支持的手动动作

后台当前支持以下动作别名：

- `cleanup-stale-live`
- `archive-room-stale`
- `merge-continuity`
- `consolidate-recent`
- `fix-orphaned`
- `delete-empty`
- `rebuild-missing`

同时也兼容旧的驼峰命名动作，便于历史脚本平滑过渡。

## 旧接口兼容策略

为了兼容既有脚本和排障流程，历史维护接口仍然保留，例如：

- `POST /api/rooms/:id/archive_stale`
- `POST /api/maintenance/merge_sessions`
- `POST /api/fix-orphaned-events`
- `POST /api/delete-empty-sessions`
- `POST /api/rebuild-missing-sessions`

但这些接口现在统一：

- 要求管理员鉴权
- 转发到 `sessionMaintenanceService` 执行
- 自动写入同一套运维日志

## 安全边界

本次改造特别收敛了维护接口的权限边界：

- 后台场次运维接口全部走 `/api/admin/*`
- 历史维护接口补齐 `authenticate + requireAdmin`
- 返回数据以概览和结果摘要为主，不向非管理员开放内部日志与修复结果

## 管理面板结构

管理后台新增“场次运维”菜单，包含四个区域：

1. 概览卡片
   - 最近 24 小时任务数、成功率、待归档数量、扫描窗口
2. 配置表单
   - 按“续场/归档”“陈旧 LIVE”“碎片合并”“日志保留”分组
3. 手动执行
   - 一键执行清理、修复、合并、单房间归档
4. 最近日志
   - 支持按任务、状态、条数筛选

## 维护建议

- 如果出现“长时间 LIVE 未归档”，优先看“陈旧 LIVE 清理”日志
- 如果出现“同一场被拆成多场”，优先看“断线续场”和“延迟归档”日志
- 如果怀疑数据跨场污染，先对单房间执行 `archive-room-stale`，再看最新日志结果
- 若后台调整了窗口参数，建议观察下一轮调度日志是否按新参数生效
