# 房间ID迁移与合并机制

## 背景

后台房间列表支持管理员手动更新房间 ID，用于处理主播改名、旧房间 ID 录错、同一房间产生多个 ID 等情况。

原有实现只支持“目标 ID 不存在”的简单迁移，且只更新 `room / event / session` 三类数据。目标 ID 已存在时会直接报错，无法在后台完成合并；同时录制任务、AI 任务、分钟统计、用户房间副本等关联数据会残留在旧房间 ID 下。

## 目标

- 支持管理员把旧房间 ID 迁移到一个全新的目标 ID。
- 当目标房间 ID 已存在时，先明确提示“将发生合并”，二次确认后再真正执行。
- 尽量保证迁移或合并完成后，房间详情、录制历史、AI 工作记录和统计缓存保持一致。
- 防止普通成员误触或直接调用接口改写全局房间数据。

## 方案

### 1. 路由权限

接口 `POST /api/rooms/:id/rename` 调整为：

- 必须登录
- 必须是管理员
- 必须具备 `session_maintenance.manage`

这样“更新房间 ID / 合并房间”与其他房间运维动作保持同一权限边界。

### 2. 二次确认机制

接口请求体为：

```json
{
  "newRoomId": "target_room_id",
  "mergeExisting": false
}
```

默认先只做目标 ID 探测：

- 若目标 ID 不存在：继续执行迁移
- 若目标 ID 已存在：返回 `409`，并带上 `code=TARGET_ROOM_EXISTS`
- 若同一房间已有迁移任务在执行：返回 `409`，并带上 `code=ROOM_RENAME_IN_PROGRESS`
- 若原房间已被上一个请求迁走，但目标房间已经存在：返回 `409`，并带上 `code=ROOM_ALREADY_MIGRATED`

前端收到该响应后，必须再次明确提示“将把旧房间数据合并到目标房间”，只有用户再次确认后，才带 `mergeExisting=true` 重试请求。

### 3. 后台任务化

房间迁移/合并不再作为同步接口直接执行，而是接入 `admin_async_job`：

- 前端提交后立即返回 `202`
- 后端创建 `maintenance.rename_room` 任务
- maintenance worker 或 web fallback 在后台消费任务
- 监控页右下角提供管理员专用浮动任务入口，展示排队、执行中、完成、失败和当前步骤
- 任务面板仅管理员可见，且对管理员共享最近任务，避免复用已有任务时面板里看不到进度
- 共享可见仅限 `maintenance.rename_room`；其它后台任务仍沿用各自权限边界，不通过该面板暴露

这样可以避免管理员在页面原地长时间等待，也能减少重复点击导致的“晚失败”体验。

### 4. 执行顺序

真正执行迁移或合并前，会先停止旧房间的采集与录制，避免迁移过程中仍有新事件继续写回旧 `room_id`。

迁移逻辑在数据库事务中完成，主要包括：

- 锁定源房间和目标房间记录
- 合并目标房间配置字段
- 合并 `user_room` 关联，避免同一用户在目标房间下留下重复副本
- 迁移 `event / session / session_summary`
- 迁移 `recording_task / highlight_clip`
- 迁移 `ai_work_job / user_ai_analysis / session_ai_review`
- 更新 `user_stats.top_room_id`
- 清理旧房间记录与旧缓存

### 5. 缓存与统计重建

以下统计类数据不做“行级拼接合并”，而是采用“删缓存 + 重建”的方式：

- `room_stats`
- `room_stats_dirty_queue`
- `user_room_stats`
- `user_room_stats_meta`
- `room_minute_stats`

事务提交后会：

- 重新写入目标房间 `room_minute_stats`
- 将目标房间加入 `room_stats` 脏队列
- 异步刷新目标房间 `user_room_stats`
- 异步刷新全局 `user_stats`

这种方式实现更稳妥，能减少主键冲突和局部统计遗漏。

### 6. 任务心跳与异常恢复

房间迁移 / 合并任务增加了执行心跳：

- 后台执行过程中会定期刷新 `admin_async_job.updated_at`
- 同步写回当前步骤与进度百分比，避免前端长时间看到静止的“执行中”

当服务重启、worker 崩溃或进程被杀死时，旧任务可能停留在 `processing`。现在会在后台轮询前先做陈旧任务恢复：

- 超过阈值仍未更新的 `processing` 任务，会自动重置为 `queued`
- 任务步骤会显示“任务执行中断，正在重新排队”
- 随后的轮询会继续捞起并重新执行，不需要管理员手工干预

当前运行配置如果未启用独立 `maintenance worker` 或 `stats worker`，Web 进程也会承担兜底轮询：

- 启动后延迟一次扫描
- 之后固定间隔继续处理 `maintenance` / `stats` 队列
- 因此即使没有独立 worker，管理员在右下角任务面板中也能持续看到进度更新和最终结果

## 影响范围

- 管理员房间列表中的“更新房间ID/迁移数据”按钮
- 管理员监控页右下角“房间迁移 / 合并”浮动任务入口
- 接口 `POST /api/rooms/:id/rename`
- 接口 `POST /api/admin/async-jobs/room-rename`
- 接口 `GET /api/admin/async-jobs` 与 `GET /api/admin/async-jobs/:id`
- 房间相关统计缓存与录制任务归属
- 后台房间运维流程

## 风险与约束

- 合并属于不可逆操作，因此必须保留显式二次确认。
- 迁移期间会先停止旧房间采集，短时间内属于预期行为。
- `user_stats` 采用异步刷新，极短时间内用户榜单可能看到旧缓存，随后会自动修正。
- 用户点击房间列表刷新按钮时，前端会带 `forceFresh=1` 重新拉取一次房间列表，绕过短时缓存，避免首屏和自动回刷误触发慢路径。
- 为避免用户重复点击造成“等待很久后才提示源房间不存在”，前端会对同一房间的迁移操作做防重复提交，后端也会对并发迁移直接返回明确冲突状态。
- 若服务在任务执行中被异常中断，任务会短暂显示重新排队，随后自动恢复执行；不再长期停留在假“执行中”状态。
