# 支付链路说明

## 背景

系统当前支持余额充值、后台支付通道管理、上游回调入账、固定码人工确认，以及移动端辅助确认。

这条链路同时涉及用户端、后台、第三方支付渠道和订单安全边界，后续修改时必须统一参考本说明。

## 支付通道

### 固定码

- 支持微信固定码、支付宝固定码。
- 后台可分别上传 1 张图片。
- 每个固定码通道都可单独配置最低金额和最高金额。
- 固定码订单默认为人工确认模式，不依赖上游自动回调。

### 富通支付

- 支持支付宝、微信两个支付方式。
- 每个支付方式都可独立启用，并配置单独的充值金额范围。
- 后台可配置：接口地址、商户号、密钥、异步通知地址、同步跳转地址、支付链接打开方式。
- 支付链接打开方式支持：
  - `qrcode`：前台展示二维码
  - `redirect`：前台直接在新窗口打开支付页

### BEPUSDT

- 作为虚拟币支付通道。
- 当前使用单一交易类型配置，并支持最低金额、最高金额设置。
- 后台可配置：接口地址、认证 Token、签名密钥、异步通知地址、支付链接打开方式。
- 若后台设置为 `redirect`，前台不展示二维码区，仅新窗口打开支付链接。
- 若后台设置为 `qrcode`，前台优先把返回的支付链接自行生成为二维码。

### PushPlus

- 仅用于固定码订单的移动确认通知，不参与自动支付。
- 后台可配置：是否启用、API 地址、Token、Channel。
- 当前默认 API 地址为 `https://www.pushplus.plus/batchSend`。
- 当前默认 Channel 为 `app`。

## 用户充值流程

### 1. 读取可用通道

用户端通过 `GET /api/payment/recharge/options` 获取当前可用充值通道。

返回结果遵循以下规则：

- 只返回后台已启用且配置完整的通道。
- 只返回前台实际需要的字段，例如标题、金额范围、打开方式。
- 若所有通道都未启用，前台显示“联系管理员充值”。

### 2. 创建充值订单

用户端通过 `POST /api/payment/recharge/create` 创建充值订单。

创建时会执行：

- 充值金额格式校验
- 通道级金额范围校验
- 通道启用状态校验
- 根据通道类型写入订单元数据

订单统一写入 `payment_records` 表。

### 3. 前台展示支付结果

根据订单元数据和通道打开方式，前台展示不同结果：

- 固定码：显示收款码图片，允许用户点击“完成支付”
- 二维码模式：展示二维码
- 跳转模式：展示支付页按钮，并在新窗口打开支付链接

## 固定码人工确认流程

### 流程说明

1. 用户创建固定码订单。
2. 用户完成转账后，在前台点击“完成支付”。
3. 后端生成 72 小时有效的移动确认签名链接。
4. 后端通过 PushPlus 把订单确认消息推送给管理员。
5. 管理员在手机端打开确认页。
6. 管理员点击“标记已支付”。
7. 系统执行与后台“标记已支付”相同的入账逻辑。

### 相关接口

- `POST /api/payment/orders/:orderNo/manual-review-request`
- `GET /api/payment/manual-review?token=...`
- `POST /api/payment/manual-review/mark-paid`

### 安全约束

- 移动确认链接使用 `JWT_SECRET` 派生签名，不在数据库保存明文令牌。
- 令牌用途固定为 `payment_manual_review`。
- 令牌过期时间为 72 小时。
- 移动确认页只返回订单确认所需最小字段。

## 自动回调流程

### 富通支付回调

- 回调入口：`/api/payment/notify/futong`
- 支持从查询参数、文本体、JSON 文本中解析回调数据。
- 只有校验通过且交易状态成功时，才会触发订单入账。
- 处理成功后返回纯文本 `success`。

### BEPUSDT 回调

- 回调入口：`/api/payment/notify/bepusdt`
- 根据渠道文档解析通知参数与签名。
- 只有订单匹配、金额通过、签名通过、状态成功时，才允许入账。
- 处理成功后返回渠道要求的成功响应。

## 后台能力

### 支付管理

后台支付管理页负责维护：

- 最低充值金额
- 快捷充值金额
- 固定码图片与金额范围
- 富通支付配置、金额范围、通知地址、打开方式
- BEPUSDT 配置、金额范围、通知地址、打开方式
- PushPlus 通知配置

配置接口：

- `GET /api/admin/payment/config`
- `PUT /api/admin/payment/config`

### 订单管理

后台订单管理支持：

- 查看订单列表
- 手动标记支付成功
- 取消订单
- 查看提交到上游的请求 URL、请求参数和上游响应

这些上游调试信息只允许管理员查看。

## 数据安全边界

### 用户接口

用户侧接口必须使用显式白名单返回字段，禁止直接暴露原始订单行或原始元数据。

用户侧不得返回以下内容：

- `channelRequest`
- `upstream`
- `notifyPayload`
- 回调地址
- 上游签名
- 渠道密钥
- 原始支付方式配置
- 其他网关调试字段

### 管理员接口

管理员可以查看完整订单调试信息，但只应在后台页面或管理员 API 中暴露。

## 请求体限制

支付配置可能包含 Base64 图片，体积明显大于普通 JSON 请求。

当前策略：

- 全局请求体限制使用 `20mb`
- 支付相关路由不再额外设置更小的 JSON 解析限制
- 避免后台保存固定码图片时再次触发 `PayloadTooLargeError`

## 后续建议

- 若生产环境存在反向代理多域名访问，建议增加“支付公共基础地址”后台配置，避免移动确认链接取错域名。
- 若后续接入更多支付渠道，应继续沿用“管理员全量序列化 / 用户最小白名单序列化”的模式。
- 若固定码订单量继续增加，可在后台补充专用的人工审核队列与通知重试机制。
