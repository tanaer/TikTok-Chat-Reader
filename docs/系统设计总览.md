# 系统设计总览

## 目标

本系统用于管理 TikTok 直播房间、沉淀事件数据，并为运营人员提供房间分析、用户分析、套餐订阅、支付充值和后台管理能力。

## 主要组成

### 1. 在线服务层

- `server.js` 提供房间列表、房间详情、分析接口、录制接口和 Socket/HTTP 入口。
- `routes/` 提供认证、用户中心、订阅、支付、后台管理等业务路由。
- `middleware/` 处理登录态、管理员权限和配额校验。

### 2. 数据层

- `room`：系统级房间主表。
- `user_room`：用户与房间的关联表，支持软删除和首次加入时间。
- `event`：直播事件明细表，是系统里体量最大、最关键的数据表。
- `room_stats`、`user_stats`、`global_stats`：聚合缓存表，用于提升分析接口性能。
- `users`、`user_subscriptions`、`subscription_plans`、`payment_records`：SaaS 用户、套餐和支付相关表。

### 3. 前端层

- `public/room_list.js`：前台房间列表与房间操作入口。
- `public/user-center.html`：用户中心。
- `public/tanaer.html` + `public/js/admin.js`：后台管理界面。

## 核心业务流

### 房间管理

1. 用户添加直播间。
2. 系统检查套餐配额和每日加房次数限制。
3. 写入 `user_room`，必要时补建 `room` 主记录。
4. 房间新增或重新启用监控后，后台会立刻触发一次 `AutoRecorder` 建连检查，不再完全依赖下一轮定时扫描。
5. 前台房间列表从系统房间和用户关联数据中组合出展示结果。

### 数据采集

1. `auto_recorder.js` 通过 `TikTokConnectionWrapper` 建立直播连接。
2. 当系统存在可用 Euler Key 时，房间 ID 与直播状态直接走 Euler 解析，不再先尝试 TikTok HTML 页面和公开 API。
3. 若 Euler 房间解析返回 429，运行时会记录限流日志、把当前 Key 放入冷却，并自动轮换到下一把 Key。
4. 如果 `room` 表中已缓存 `numeric_room_id`，系统会优先直接使用该 numeric roomId 建立 WebSocket 连接，避免在 TikTok HTML/API 暂时失真时整条采集链路被卡死。
5. 连接成功后，保活逻辑优先使用 `webcast/room/check_alive` 做数字房间号判活，不再把 `fetchIsLive()` 失败或短时间无事件直接当成离线，避免安静直播间被误踢。
6. 当已知 numeric roomId 连接成功，但 TikTok 的房间详情接口仍返回空值或异常时，系统会先保持连接，等待 `check_alive=true` 或首条实时事件来确认直播，不再把“已连上”直接当成“已开播”。
7. 系统内部明确区分两类标识：
   - `uniqueId`：房间列表、详情页和业务侧使用的主播标识。
   - `numeric roomId`：TikTok WebSocket、`check_alive` 和录播使用的真实直播间号。
8. Socket 推送给前端时，会同时传 `uniqueId` 和 `roomId`，前端展示连接日志时只把 `uniqueId` 当房间标识，`numeric roomId` 仅作为调试信息附带显示，避免把用户 ID 误当真实房间号。
9. 自动巡检除了负责新建连接，也会识别 `activeConnections` 中 WebSocket 已断开的陈旧连接；这类失活连接会先被回收，再重新纳入下一轮自动建连，避免出现“后台不再重连，只能手动点连接恢复”的卡死状态。
10. 自动巡检不再直接把“发起 WebSocket 连接”当作探活。它会先走轻量预探活：
   - 若缓存的 `numeric roomId` 仍能通过 `check_alive`
   - 或 TikTok live 页快照返回 `status=2`
   才进入真正的 WebSocket 建连。
11. 预探活阶段如果发现当前场次的 `numeric roomId` 已变化，会先刷新到 `room.numeric_room_id`，避免后台长期拿旧房间号探活，造成“明明开播却一直判离线”。
12. 自动巡检与手动连接在成功拿到 numeric roomId 后，都会统一触发自动录制检查；只要房间配置开启了自动录制，且当前未在录制中，就会直接拉起录播，避免“列表显示恢复但录制未启动”的分叉行为。
13. 自动巡检会限制并发预探活数量、拉开连接启动间隔，并在出现 `Too many connections started` 之类上游限流时进入全局冷却，避免房间数量上来后把底层连接通道自己打爆。
14. 预探活并发不再固定锁死在极低值，而是按“当前可用 Key 数 + 安全上浮”动态计算，并保留可配置上限；真正的 WebSocket 启动仍通过连接启动间隔做限速，因此“多 Key”主要提升预探活吞吐和容错，不会粗暴放大真实建连风暴。
15. 自动巡检每轮开始前会先刷新一次 Euler Key 池和代理配置；Web 服务启动完成数据库初始化后，也会先装载 `euler_api_keys` 表，避免进程长时间停留在旧的环境变量 Key 集合。
16. 遗留事件清理与场次合并不再对每个待扫房间都提前执行，而是只在预探活确认“当前确实在线、准备建连”后执行，避免大房间池下扫描吞吐被运维清理逻辑拖慢。
17. 如果某房间在进程内残留了 `wsConnected=true` 但长期没有事件、也没有通过 `check_alive` 确认的“不可用连接”，自动巡检和手动连接都会先回收这条占坑连接，再重新建连，避免房间长期卡在“看似已连接、实际不再自愈”的假状态。
18. 手动连接链路也会先做和自动巡检同级别的预探活，必要时刷新 `numeric_room_id` 后再建连，不再继续盲目复用过期房间号。
19. 定时扫描下一轮触发时间会扣除上一轮实际执行耗时，避免在房间规模扩大后出现“配置每 5 分钟扫描，实际却越来越慢”的节拍漂移。
20. 自动巡检会从历史 `session` 场次里提取每个房间的高概率开播时间窗；时间窗附近的房间提高扫描优先级，远离时间窗的冷房间自动退避扫描，减少大房间池下无效轮询。
21. 对历史样本不足的新房间，系统不会强行降频，而是继续按普通策略扫描，避免新房间因为没有画像被长期错过。
22. 后台“场次运维”提供自动监控诊断面板，直接展示 Key 池负载、全局建连冷却、房间级扫描优先级、时段退避和近期失败记录，帮助定位“为什么没自动监控到”。
23. 该诊断面板只暴露运行态计数、房间 ID 和掩码后的 Key 样本，不返回原始 Euler Key、Cookie 或上游敏感字段。
24. 对“探活经常被上游拦住，但近期使用缓存 `numeric_room_id` 直连成功过”的房间，系统会建立短期动态画像，在 TTL 内提高其扫描和兜底直连优先级；这类画像会随着失败、下播或超时自动衰减，不做永久白名单。
25. 获取到 `roomId / wsUrl / cursor` 后，系统再连接 TikTok WebSocket 持续接收弹幕、礼物、点赞和在线人数事件。
26. 事件落入 `event` 表，供实时展示、聚合统计和会话归档复用。

### 直播状态判定优先级

1. `auto_recorder.js` 中的活跃连接是实时状态的第一来源，但只有满足以下任一条件才会被标记为“直播中”：
   - `webcast/room/check_alive` 返回 `alive=true`
   - 已收到首条实时事件，并持续有近期事件流入
2. 仅仅完成 WebSocket 连接，不代表房间已经确认在播。连接成功只说明“采集通道已建立”，不能直接推导成“房间列表应显示直播中”。
3. `services/liveStateService.js` 作为短周期共享缓存，在以下两种场景会参与回退：
   - 本机没有该房间的活跃连接记录
   - 本机虽然仍保留该房间的跟踪记录，但 WebSocket 已失活
4. `event.session_id IS NULL` 的近期未归档事件，只作为最后兜底，避免单机切换、短时重启或陈旧连接残留时短暂丢失直播标记；只有当本机存在“已确认可用”的 live 连接时，才会压过这层兜底。
5. 断线下线采用双重确认：
   - 若 `check_alive=false`，先进入确认窗口
   - 确认窗口内仍持续无事件，才真正标记离线并归档

### 数据库启动保护

1. `db.js` 启动时会先读取 `settings._db_schema_bootstrap_version`。
2. 当数据库结构已完成当前版本引导后，后续 Web 进程重启不会重复执行整套 `ALTER TABLE` / `CREATE INDEX` 初始化语句。
3. 这样可以避免服务重启时再次对 `event` 等大表申请排他锁，降低直播采集中途因为初始化 DDL 与在线查询互相阻塞的风险。

### 数据分析

1. 直播事件进入 `event`。
2. 在线接口读取缓存表获取长期指标。
3. 需要实时值时，再补充计算当前直播中的短期指标。
4. 详情页和分析页在不同粒度上消费这些数据。

### 后台管理

1. 管理员在后台查看用户、订单、套餐、支付配置和系统设置。
2. 管理员可调整用户配额、处理充值订单、配置 AI 通道。
3. 后台“系统文档”页读取 `docs/` 目录下的 Markdown 文档，帮助管理员了解系统设计。

## 设计原则

- 在线接口优先使用缓存和聚合表，避免直接扫描大表。
- 用户态与管理员态严格分离，后台接口默认只对管理员开放。
- 用户房间数据与系统房间数据分层保存，方便权限控制和软删除。
- 所有系统设计、架构调整和关键机制变更，应同步写入 `docs/`。
- 支付相关能力、通道配置、回调与人工确认流程，见 `docs/支付链路说明.md`。
- 用户中心站内通知、到账提醒与轮询机制，见 `docs/站内通知机制.md`。
