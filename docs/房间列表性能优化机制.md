# 房间列表性能优化机制

## 问题背景

房间列表相关接口主要是：

- `/api/rooms/stats`
- `/api/rooms`

当 `event` 表持续增长后，这两个接口会受到两类压力影响：

1. 房间列表自身的查询与聚合开销。
2. Web 进程内其他重型统计任务对数据库的挤占。

## 本次优化内容

### 1. 合并用户房间上下文查询

房间列表接口原本会分别查询：

- 用户可见房间列表
- 用户房间别名与首次加入时间

现在改为单请求内只查一次 `user_room`，再复用结果生成：

- `roomFilter`
- `userRoomData`
- `dataStartTimes`

这样可以减少重复 SQL 和无意义的等待。

### 2. 给列表接口增加短缓存

对以下接口增加了短缓存：

- `/api/rooms/stats`
- `/api/rooms`

缓存特征：

- 默认 TTL：10 秒
- 维度：按用户、分页、搜索词、排序方式区分
- 在新增房间、删除房间、编辑房间、重命名房间、停止房间时自动失效

这样能显著降低房间列表频繁刷新带来的重复查询成本。

### 3. 搜索增加精确匹配快路径

当搜索词像 `pinkpig606` 这样的房间 ID 时，系统先尝试精确匹配。

如果命中，直接走 `room_id = ?`，避免先走 `%keyword%` 模糊搜索。

只有精确匹配未命中时，才回退到模糊搜索。

### 4. 实时聚合只计算当前页直播房间

`/api/rooms/stats` 原先会对当前页所有房间都尝试从 `event` 表做实时聚合。

现在改为：

- 只对当前页“正在直播”的房间计算实时指标
- 离线房间直接使用缓存表和已有静态指标

这样可以避免对大 `event` 表做大量无效扫描。

### 5. `room_stats` 缺失时按页自愈

房间列表中的历史指标依赖 `room_stats` 聚合表。

如果某些房间已经有 `event` / `session` 数据，但 `room_stats` 缺行或落后于最新事件，接口会出现：

- 月礼物为 `0`
- 总礼物为 `0`
- TOP 占比为 `0`
- 最近场次时间为空

现在 `/api/rooms/stats` 在查询当前页房间后，会先检查这些房间的 `room_stats` 是否缺失或过期：

- 缺失 `room_stats` 行时自动补算
- `room_stats.updated_at` 早于最新 `event.timestamp` 时自动补算
- `room_stats.last_session_time` 落后于最新 `session.created_at` 时自动补算

补算范围只限当前页房间，不做整表扫描，避免把单次房间列表请求重新变成重型任务。

### 6. 无 stats worker 时，Web 进程默认兜底刷新

之前 `room_stats` / `user_stats` 的启动预热和周期刷新默认关闭，适合已经部署独立 stats worker 的环境。

但如果线上没有开启 `ENABLE_WORKER_STATS`，同时又没有显式打开：

- `ENABLE_PERIODIC_STATS_REFRESH`
- `ENABLE_STARTUP_STATS_WARMUP`

那么聚合表会长期不刷新，房间列表就会越来越多地显示历史值为 `0`。

现在改为：

- 如果启用了独立 stats worker，仍由 worker 负责
- 如果没有启用 stats worker，Web 进程默认承担启动预热与周期刷新
- 如果运维显式配置了这两个环境变量，仍以显式配置为准

### 7. 默认关闭 Web 进程内的重型全量刷新

以下任务会扫描大表：

- `room_stats` 全量刷新
- `user_stats` 全量刷新

在有独立 stats worker 的部署里，默认不在 Web 进程里自动执行：

- 启动预热关闭
- 周期刷新关闭

如需显式覆盖，可使用环境变量：

- `ENABLE_PERIODIC_STATS_REFRESH=true`
- `ENABLE_STARTUP_STATS_WARMUP=true`

## 配套索引

本次补充了以下索引：

- `room(updated_at DESC)`
- `user_room(user_id, room_id) WHERE deleted_at IS NULL`
- `user_room(user_id, first_added_at) WHERE deleted_at IS NULL`

## 当前建议

- 在线 Web 进程只负责接口响应。
- 大统计任务放到独立 worker 或系统 cron。
- 如果后续房间列表访问量继续上升，可继续把“当前直播实时指标”做成独立增量表。
