# 房间列表性能优化机制

## 问题背景

房间列表相关接口主要是：

- `/api/rooms/stats`
- `/api/rooms`

当 `event` 表持续增长后，这两个接口会受到两类压力影响：

1. 房间列表自身的查询与聚合开销。
2. Web 进程内其他重型统计任务对数据库的挤占。

## 本次优化内容

### 1. 合并用户房间上下文查询

房间列表接口原本会分别查询：

- 用户可见房间列表
- 用户房间别名与首次加入时间

现在改为单请求内只查一次 `user_room`，再复用结果生成：

- `roomFilter`
- `userRoomData`
- `dataStartTimes`

这样可以减少重复 SQL 和无意义的等待。

### 2. 给列表接口增加短缓存

对以下接口增加了短缓存：

- `/api/rooms/stats`
- `/api/rooms`

缓存特征：

- 默认 TTL：10 秒
- 维度：按用户、分页、搜索词、排序方式区分
- 在新增房间、删除房间、编辑房间、重命名房间、停止房间时自动失效

这样能显著降低房间列表频繁刷新带来的重复查询成本。

### 3. 搜索增加精确匹配快路径

当搜索词像 `pinkpig606` 这样的房间 ID 时，系统先尝试精确匹配。

如果命中，直接走 `room_id = ?`，避免先走 `%keyword%` 模糊搜索。

只有精确匹配未命中时，才回退到模糊搜索。

### 4. 实时聚合只计算当前页直播房间

`/api/rooms/stats` 原先会对当前页所有房间都尝试从 `event` 表做实时聚合。

现在改为：

- 只对当前页“正在直播”的房间计算实时指标
- 离线房间直接使用缓存表和已有静态指标

这样可以避免对大 `event` 表做大量无效扫描。

### 5. `room_stats` 改为后台脏队列修复

房间列表中的历史指标依赖 `room_stats` 聚合表。

如果某些房间已经有 `event` / `session` 数据，但 `room_stats` 缺行或落后，接口会出现：

- 月礼物为 `0`
- 总礼物为 `0`
- TOP 占比为 `0`
- 最近场次时间为空

现在不再在 `/api/rooms/stats` 请求链路里同步补算 `room_stats`，避免已登录用户打开房间列表时触发重查询。

改为把受影响房间写入 `room_stats_dirty_queue`，由后台批处理修复，主要触发点包括：

- 正常下播归档完成
- 陈旧 LIVE 事件归档完成
- 缺失场次重建 / 孤儿事件修复 / 场次合并完成
- 房间重命名迁移完成
- 新房间创建后需要初始化缓存行

后台处理时会按批次取出脏房间，只对这些房间执行 `refreshRoomStats({ roomIds })` 局部刷新，不做整表扫描。

### 6. 轻量兜底扫描只补缺失房间

除业务触发外，后台还会定期扫描：

- `room` 表中存在
- `room_stats` 中缺失
- 且尚未进入脏队列

的房间，把它们补进脏队列。

这个兜底只负责“缺行修复”，不在房间列表请求中同步执行，也不默认做重型全表刷新。

### 7. 默认关闭 Web 进程内的重型全量刷新

以下任务会扫描大表：

- `room_stats` 全量刷新
- `user_stats` 全量刷新

默认仍不在 Web 进程里自动执行重型全量刷新：

- 启动预热关闭
- 周期刷新关闭

如需显式执行全量刷新，可使用环境变量：

- `ENABLE_PERIODIC_STATS_REFRESH=true`
- `ENABLE_STARTUP_STATS_WARMUP=true`

## 配套索引

本次补充了以下索引：

- `room(updated_at DESC)`
- `user_room(user_id, room_id) WHERE deleted_at IS NULL`
- `user_room(user_id, first_added_at) WHERE deleted_at IS NULL`

## 当前建议

- 在线 Web 进程只负责接口响应。
- `room_stats` 修复尽量走事件驱动入队 + 后台批处理。
- 大统计任务放到独立 worker 或系统 cron。
- 如果后续房间列表访问量继续上升，可继续把“当前直播实时指标”做成独立增量表。

## 用户分析列表补充

### 首屏轻量化

用户分析列表接口 `/api/analysis/users` 现已支持按需裁剪重字段：

- `includeTopGifts=false`
- `includeHistoryMeta=false`

当列表首屏只需要主表字段时，前端会显式传这两个参数，服务端将跳过：

- 当前页用户的礼物明细聚合
- 历史账号 / 历史昵称聚合

这样可以避免首屏请求被 `event` 表上的附加聚合拖慢。

### 明细延后补齐

新增接口 `/api/analysis/users/enrichment`，用于按用户批量补充：

- `topGifts`
- `historyUniqueIds`
- `historyNicknames`
- `historyAliasCount`

约束如下：

- 仍沿用当前登录用户的 `roomFilter` 权限边界
- 仅补充请求中的用户 ID
- 默认走短 TTL Redis 分析缓存

前端策略改为：

1. 先请求轻量版 `/api/analysis/users`
2. 先渲染表格主信息
3. 再异步请求 enrichment 补齐“常刷礼物”和“历史账号”按钮

这样可以把“首屏可见时间”和“附加信息完备时间”拆开，降低首次进入用户分析页的等待感知。

### 再次收敛

后续又发现，如果 enrichment 一次携带几十个用户并同时请求：

- `includeTopGifts=true`
- `includeHistoryMeta=true`

仍会对 `event` 表造成明显压力，表现为：

- “常刷礼物”长时间 loading
- 历史账号聚合接口等待很久

因此当前策略继续收敛为：

- 列表页默认只批量补 `topGifts`
- `historyMeta` 改为点击“历史账号”时再按单用户查询
- enrichment 前端批次缩小，并改为顺序补齐，避免瞬时并发打爆数据库

同时数据库增加更贴近该场景的索引：

- `event(user_id) WHERE type = 'gift'`
- `event(user_id, unique_id, timestamp DESC) WHERE unique_id IS NOT NULL AND unique_id <> ''`
- `event(user_id, nickname, timestamp DESC) WHERE nickname IS NOT NULL AND nickname <> ''`

历史账号查询也改成 `DISTINCT ON (...) ORDER BY timestamp DESC` 的写法，以便更稳定命中新索引。

### 预聚合补强

为了进一步降低管理员 / 全局用户分析列表的 enrichment 开销，`user_stats` 增加了：

- `top_gifts_json`

该字段在 `refreshUserStats()` 时一并刷新，保存用户全局维度下的前 6 个常刷礼物。

读路径策略改为：

- 无 `roomFilter` 时优先直接读取 `user_stats.top_gifts_json`
- 只有成员态带房间过滤时，才按房间范围实时聚合 `topGifts`
- 如果缓存字段尚未完成刷新，读路径会对缺失用户做小批量回退查询，避免功能直接失效

这样可以把管理员视角最常见的 enrichment 请求，从“每次都扫 `event` 聚合”改成“优先读缓存，缺失再兜底”。

### 成员房间级缓存改为后台预热

成员态用户分析列表天然带 `roomFilter`，即便首屏已经关闭：

- `includeTopGifts`
- `includeHistoryMeta`

如果仍直接走 `event` 实时聚合，在房间数据量较大时首开仍会偏慢。

为此新增了两张房间级缓存表：

- `user_room_stats`
- `user_room_stats_meta`

缓存内容覆盖：

- 房间内用户累计礼物
- 聊天次数
- 玫瑰 / 刀礼物统计
- 房间维度 `top_gifts_json`
- 最近活跃时间

当前读路径规则为：

1. 仅当满足以下条件时尝试走房间级缓存：
   - 存在 `roomFilter`
   - 未使用 `activeHour`
   - 未使用 `dataStartTimes`
2. 如果目标房间缓存已存在，直接从 `user_room_stats` 聚合分页结果
3. 如果目标房间缓存尚未建立，不再在请求链路里同步执行 `refreshUserRoomStats(roomIds)`
4. 改为异步调度后台预热，并让本次请求立即回退到旧的实时查询逻辑

这样做的目的有两个：

- 首次进入成员用户分析页时，不会因为“顺手建缓存”而把请求阻塞到几十秒
- 第二次及后续请求可以直接命中房间级缓存，显著缩短成员态列表查询时间

为了避免重复预热，同一批房间在 Web 进程内会做 `in-flight` 去重，只允许一个后台刷新任务执行。
