# 登录验证码机制

## 当前状态

登录页已暂时停用滑块/拼图验证码，原因是移动端交互体验较差，存在无法正常完成验证的问题。

当前登录保护改为：

- 默认登录不需要验证码
- 同一 IP 连续密码错误 3 次后，后续登录会要求输入图形验证码
- 图形验证码沿用注册页同款的数字图形验证码机制
- 用户点击登录后，如命中验证码条件，再弹出验证码输入层
- 验证通过后自动继续刚才的登录请求
- 登录成功后会清空该 IP 的连续失败计数

## 组成

### 前端

- `public/login.html`
  - 登录提交后按后端返回决定是否弹出验证码
- `public/js/login-captcha-flow.js`
  - 登录专用图形验证码弹窗封装
  - 复用注册页同类交互：加载图形验证码、换一张、输入 5 位数字、确认继续

### 后端

- `services/captchaService.js`
  - 负责生成和校验数字图形验证码
  - 当前同时支持 `send-code` 与 `login` 两类用途
- `services/sliderCaptchaService.js`
  - 当前仅保留“按 IP 连续失败计数”的能力
  - 用于决定登录是否需要验证码
- `routes/auth.js`
  - `GET /api/auth/login-captcha`
  - `POST /api/auth/login` 中按条件校验图形验证码

## 数据流

1. 用户点击登录
2. 后端判断当前 IP 是否达到验证码阈值
3. 若未达到阈值，则直接校验用户名/邮箱与密码
4. 若达到阈值，则返回 `CAPTCHA_REQUIRED`
5. 前端弹出图形验证码输入层
6. 前端请求 `GET /api/auth/login-captcha` 获取图片和 `captchaToken`
7. 用户输入验证码后，前端自动重试原登录请求，并携带 `captchaToken` 与 `captchaAnswer`
8. 后端验证成功后继续执行登录逻辑

## 说明

- `jigsaw` 相关登录交互已暂时停用，不再作为当前登录主流程的一部分
- 后续如需恢复拼图验证码，应先单独评估移动端兼容性与交互成本
