# 登录滑块验证码机制

## 目标

为系统提供一套可复用的滑块验证码能力，先在登录页启用，后续可扩展到注册、敏感操作和高风险接口。

## 选型

当前前端滑块交互基于 `ArgoZhang/SliderCaptcha` 的开源实现，本地托管其脚本文件，避免依赖外部 SaaS。

选型原因：

- 交互成熟，支持桌面端和移动端拖动
- 支持把拖动轨迹上送服务端做二次校验
- 无需接入第三方账号、AppId 或远程厂商服务
- 适合当前静态页面 + Node.js 后端架构快速复用

## 组成

### 前端

- `public/vendor/slider-captcha/longbow.slidercaptcha.js`
  - 第三方滑块核心脚本，本地托管
- `public/css/slider-captcha.css`
  - 针对当前系统样式做了隔离处理，避免第三方全局 CSS 污染现有页面
- `public/js/slider-captcha-flow.js`
  - 系统级共享封装
  - 负责初始化滑块、生成本地背景图、调用后端验证接口、缓存短期通行令牌

### 后端

- `services/sliderCaptchaService.js`
  - 校验拖动轨迹
  - 生成短时效通行令牌
  - 校验并消费通行令牌，防止重复复用
- `routes/auth.js`
  - `POST /api/auth/slider-captcha/verify`
  - `POST /api/auth/login` 中强制校验 `sliderPassToken`

## 数据流

1. 登录页加载共享滑块组件
2. 用户完成前端拼图滑动
3. 前端将滑动轨迹和耗时发送到 `/api/auth/slider-captcha/verify`
4. 服务端通过轨迹规则校验后签发短时效 `passToken`
5. 登录请求提交时携带 `sliderPassToken`
6. 服务端登录接口消费该令牌，通过后继续执行账号密码校验
7. 登录失败后前端重置滑块，要求重新验证，避免一次验证被反复撞库复用

## 安全策略

- 通行令牌默认 5 分钟过期
- 通行令牌与用途绑定，当前仅用于 `login`
- 通行令牌与请求 IP、User-Agent 绑定
- 登录接口消费后即失效，避免重放
- 轨迹必须满足最小长度、耗时范围和基础离散性校验

## 复用方式

后续页面如需接入，只需：

1. 引入 `public/css/slider-captcha.css`
2. 引入 `public/vendor/slider-captcha/longbow.slidercaptcha.js`
3. 引入 `public/js/slider-captcha-flow.js`
4. 调用 `SharedSliderCaptcha.create({ purpose, widget, status })`
5. 在业务接口中携带 `passToken`
6. 服务端在对应接口中消费该令牌

## 当前接入范围

- 已启用：登录页 `public/login.html`
- 预留复用：注册、找回密码、邮箱变更、后台高风险操作
