# AI客户分析具体 Ticket 清单

## 说明

本文把 `AI客户分析上下文引擎` 的开发计划拆成可以直接进入 Jira、禅道、GitHub Projects 或飞书项目的具体 Ticket。

默认约束如下：

1. 用户事实数据必须由系统计算，不能由 AI 编造。
2. 用户态接口必须继续做白名单返回，不能泄露内部调试字段或原始结构。
3. 本轮优先做 MVP，不一次性完成所有模型和所有变量语法。
4. 本轮优先兼容现有 `POST /api/analysis/ai`，不重做整套用户分析前端。
5. 如需重启项目，只能使用：`systemctl restart tiktok-monitor`。

## Ticket 约定

### 优先级

- `P0`：必须优先处理，直接影响本轮 MVP 能否落地。
- `P1`：高优先级，建议在 MVP 跑通后紧接着完成。
- `P2`：增强项，建议在基础链路稳定后推进。
- `P3`：扩展项，可放到下一阶段。

### 角色建议

- `BE`：后端开发
- `FE`：前端开发
- `DBA`：数据库/数据工程
- `QA`：测试
- `PM`：产品/运营

### 状态建议

每个 Ticket 建议初始状态为 `Todo`，并在系统里额外维护：

- `Blocked`
- `Ready for Dev`
- `Ready for Test`
- `Ready for Release`
- `Done`

## Phase 0：定义与边界确认

### CA0-01 梳理现有用户分析链路与返回面

- **角色**：BE
- **目标**：明确当前 `/api/analysis/ai`、缓存表、前端渲染和权限边界。
- **主要工作**：
  - 梳理 `POST /api/analysis/ai` 当前请求、缓存、扣点、返回结构。
  - 梳理 `user_ai_analysis` 与 `user.ai_analysis` 的现状用途。
  - 做一次 response-surface review，确认用户态可返回字段边界。
- **影响文件**：`server.js`、相关表结构说明文档
- **依赖**：无
- **验收标准**：
  - 输出当前链路说明。
  - 确认本轮改造不会直接暴露内部上下文原文到普通用户接口。

### CA0-02 定义 `customerContextJson` MVP 结构

- **角色**：BE / PM
- **目标**：冻结第一版上下文结构，避免边开发边变。
- **主要工作**：
  - 确认 `identity`、`metricCube`、`preference`、`models`、`signals` 五层结构。
  - 确认 MVP 必须字段和允许为空字段。
  - 确认空间维度与时间维度枚举。
- **影响文件**：`docs/plans/ai-customer-analysis/2026-03-08-AI客户分析上下文引擎开发计划.md`
- **依赖**：无
- **验收标准**：
  - 新 session 开发时不再需要反复确认字段命名。

### CA0-03 定义客户分析 Prompt 模板与输出 JSON 结构

- **角色**：PM / BE
- **目标**：明确 AI 的输入和输出契约。
- **主要工作**：
  - 确认 `AI客户分析 · 主分析流程` 默认 Prompt。
  - 确认输出 JSON 字段、限制和容错规则。
- **影响文件**：`services/aiPromptService.js`、文档
- **依赖**：CA0-02
- **验收标准**：
  - 输出结构适合前端后续结构化渲染。
  - Prompt 中明确“数值必须来自系统”。
