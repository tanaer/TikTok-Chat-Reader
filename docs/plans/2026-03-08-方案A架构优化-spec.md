# 方案A架构优化 Spec（保留 PostgreSQL 主库）

## 背景

本 Spec 面向当前 `tiktok-monitor` 系统，目标是在**不更换主数据库品类**的前提下，解决以下现实问题：

- 直播事件持续写入，`event` 已成为系统最大、最关键的热点表。
- Web 进程同时承担接口响应、定时统计、缓存和部分运维任务，职责偏重。
- 录播文件直接落本地 `downloads/`，磁盘压力高于业务库增长速度。
- 房间列表、房间详情、用户分析等能力已经开始依赖聚合缓存表，但缓存构建仍有明显“全表扫描 + 定时刷新”特征。
- 系统已经从 SQLite 迁到 PostgreSQL，但数据模型和统计实现还保留了“从原始事件临时算结果”的思路。

本 Spec 以 **2026-03-08** 的代码和现场数据为依据，默认约束如下：

1. 保留 `Node.js + Express + PostgreSQL` 主架构。
2. 尽量不改现有用户态 API 契约。
3. 优先做 1 到 3 个月内可落地的低风险优化。
4. 不在本阶段引入 `InfluxDB`、`MySQL` 或新的主事务库。

## 当前系统事实

### 1. 数据和资源现状

基于本机实测：

- 服务器规格约为 `4 vCPU / 15 GiB RAM / 150 GiB` 系统盘。
- 当前磁盘可用空间约 `39 GiB`。
- PostgreSQL 主库 `tkmonitor` 约 `27 GiB`。
- `event` 表估算约 `4,776 万` 行，总占用约 `26 GiB`。
- `event` 表中，表数据约 `9 GiB`，索引约 `18 GiB`，索引体积明显高于数据体积。
- 本地 `downloads/` 目录约 `23 GiB`，且只有约 `20` 个 mp4 文件，说明录播文件比数据库更快吃掉磁盘。

按当前增长速度粗算：

- `event` 日增长约 `55 万` 行。
- 数据库日增长约 `313 MB`。
- 录播文件实测体积约 `3.6 ~ 5.1 GB/小时`。

结论很明确：**当前系统先遇到的是单机磁盘与大表维护问题，不是 CPU 问题。**

### 2. 当前数据流

当前主数据流如下：

1. `auto_recorder.js` 从 TikTok 直播间接收 `member` / `chat` / `gift` / `like` / `roomUser` 事件。
2. 每条事件直接调用 `manager.logEvent(...)` 写入 PostgreSQL。
3. Web 层接口从 `room`、`session`、`event`、`room_stats`、`user_stats`、`global_stats` 中取数。
4. 统计缓存由 `server.js` 中的定时任务驱动刷新。
5. 录播由 `recording_manager.js` 直接写入本地 `downloads/`。

这个架构已经具备“主库 + 聚合缓存”的雏形，但数据分层还不够清晰：**原始事件、在线查询、定时统计、录播文件、实时态仍在一台机器上抢资源。**

### 3. 当前代码里的关键问题

#### 3.1 `event` 是统一明细表，但字段设计与查询需求不完全对齐

`event` 当前主字段定义在 `db.js` 中，包括：`room_id`、`session_id`、`type`、`timestamp`、`user_id`、`gift_id`、`diamond_count`、`repeat_count`、`comment`、`viewer_count` 等。

但 `manager.logEvent(...)` 现在写入的是“展开列”，并**没有持续写入 `data_json`**。同时，代码中仍有不少统计逻辑依赖 `data_json::json->>'giftName'` 一类表达式。这会带来两个问题：

- 新旧数据模型不一致，历史数据和新数据的可分析性不完全一致。
- 某些统计仍依赖 JSON 解析，既慢，也不稳定。

#### 3.2 聚合缓存表已经存在，但更新方式仍偏“全量扫描”

系统已有：

- `room_stats`
- `user_stats`
- `global_stats`

这些表方向是对的，但更新逻辑仍以批量重算为主。数据量再继续增长后，`refreshRoomStats()`、`refreshUserStats()`、`refreshGlobalStats()` 会越来越重。

#### 3.3 Web 进程仍承担定时重任务

尽管已有“默认关闭部分重刷新”的优化文档，但当前系统仍保留 Web 进程内定时刷新、会话归并、过期清理等逻辑。只要未来恢复更多定时任务，接口响应就会被大表扫描挤压。

#### 3.4 录播文件与业务数据库共用本地磁盘

这是当前最直接的风险点。只要多录几场长直播，本地磁盘就可能先被视频写满，进而连带影响 PostgreSQL。

## 目标

### 业务目标

1. 保持现有用户态和管理员态主要接口可用，不做大范围前端重写。
2. 在现有业务模型下，把系统从“单机堆叠”调整为“单主库 + 共享缓存 + 对象存储 + 独立 worker”。
3. 支撑未来 `8,000 万 ~ 1 亿+` 级事件规模，且房间列表、详情页、用户分析仍可接受。
4. 把录播文件从主机磁盘中剥离，避免磁盘先爆。

### 技术目标

1. PostgreSQL 继续作为系统唯一主事务库。
2. `event` 从单大表演进为按时间分区的大表。
3. Web 进程不再负责重型统计刷新。
4. 实时态与短缓存进入 Redis。
5. 原始事件、增量汇总、长期统计分层存储。
6. 新增字段后，统计逻辑尽量不再依赖 `data_json`。

## 非目标

本阶段明确**不做**以下事情：

1. 不把主库切换为 MySQL。
2. 不引入 InfluxDB 作为业务主存储。
3. 不在本阶段引入 Kafka、Pulsar 等消息队列。
4. 不改动支付、认证、套餐等核心业务模型。
5. 不追求“所有统计都改为毫秒级实时”。

## 方案选择

### 备选方向

#### 方向 1：继续单机 PostgreSQL，局部打补丁

优点：

- 改动最小。
- 短期能继续运行。

缺点：

- 无法根治磁盘与职责耦合问题。
- 重任务仍容易回流到 Web 进程。
- 后续扩展空间差。

#### 方向 2：方案A，保留 PostgreSQL 主库，增加 Redis、对象存储、独立 worker，并做 `event` 分区

优点：

- 改造成本可控。
- 与当前代码结构兼容度高。
- 能明显改善磁盘、缓存、统计和扩展问题。

缺点：

- 需要做一次中等规模的架构整理。
- 需要补一轮数据迁移和运维脚本。

#### 方向 3：直接引入新的分析主存储（InfluxDB / ClickHouse）

优点：

- 理论上可获得更强分析扩展性。

缺点：

- 设计、迁移和运维成本高。
- 当前瓶颈并不主要来自数据库品类不对。
- 复杂度明显超出本阶段收益。

### 结论

本 Spec 采用**方向 2**，即：

> 保留 PostgreSQL 作为唯一主事务库；把录播文件迁出本地磁盘；把短 TTL 缓存和实时态迁到 Redis；把重型定时任务迁到独立 worker；把 `event` 改造为按时间分区并增加派生汇总层。

## 目标架构

### 架构总览

系统调整为以下五层：

1. **采集层**：`auto_recorder.js` 与 TikTok 连接，持续接收直播事件。
2. **在线服务层**：`server.js + routes/ + services/`，只负责接口响应、鉴权、轻量读写、Socket 转发。
3. **缓存与实时态层**：Redis，承载房间列表短缓存、当前直播间实时状态、分布式锁。
4. **主数据层**：PostgreSQL，承载用户、支付、房间、会话、事件明细、汇总表。
5. **文件与离线任务层**：对象存储承载录播文件，worker 负责统计、上传、清理和回填。

### 核心原则

1. **PostgreSQL 只做主数据，不做一切。**
2. **Web 进程只做在线请求，不做重刷新。**
3. **对象文件不再长期占用本地系统盘。**
4. **原始事件保留，在线查询尽量走汇总或缓存。**
5. **实时功能允许秒级最终一致，不要求强一致。**

## 详细设计

### 一、主数据库角色不变，但 `event` 结构升级

#### 1.1 保留 PostgreSQL 为唯一主库

以下实体继续保留在 PostgreSQL：

- `users`
- `user_room`
- `room`
- `session`
- `event`
- `room_stats`
- `user_stats`
- `global_stats`
- `payment_records`
- `user_subscriptions`
- `settings`
- `recording_task`

原因很简单：这些数据需要事务、权限控制、后台管理和多表关联。当前系统天然是 OLTP + 轻分析混合场景，PostgreSQL 足以胜任主库角色。

#### 1.2 `event` 改造为按月分区表

建议采用：

- 分区键：`timestamp`
- 粒度：**按月分区**
- 命名：`event_YYYY_MM`

按当前规模，按月分区已经足够。如果未来单月事件量继续升高，再考虑切到按周分区。

分区后的收益：

- 最近 7 天、30 天、当月统计只扫描相关分区。
- 删除冷数据或迁移历史数据更容易。
- 索引不再全部堆在一个超大物理表上。
- `VACUUM / ANALYZE / REINDEX` 的影响范围更可控。

#### 1.3 补齐“分析真正需要的列”

当前很多统计逻辑仍依赖 JSON 字段或旧数据模型。建议在 `event` 中显式增加以下列：

- `gift_name`
- `gift_image`
- `group_id`
- `region`
- `is_admin`
- `is_super_admin`
- `is_moderator`
- `fan_level`
- `fan_club_name`

原则：**凡是会被高频筛选、聚合、排序、展示的字段，都应优先落展开列，不应长期依赖 `data_json`。**

#### 1.4 索引策略调整

本阶段不再继续“见查询就加索引”，而是做一次索引瘦身。

建议保留的主索引围绕以下查询模式设计：

- 房间最近事件：`(room_id, timestamp DESC)`
- 会话事件明细：`(session_id, timestamp DESC)`
- 房间内礼物聚合：`(room_id, type, timestamp DESC)` 或礼物专用部分索引
- 用户最近活跃：`(user_id, timestamp DESC)`
- 当前直播未归档事件：`(room_id, timestamp DESC) WHERE session_id IS NULL`

建议动作：

1. 统计现有索引使用率。
2. 删除几乎不使用但占用巨大的索引。
3. 对冷分区优先使用更轻量的时间索引方案。
4. 避免语义重复的索引长期并存。

### 二、录播文件迁移到对象存储

#### 2.1 现状问题

当前录播文件直接落到 `downloads/`，路径保存在 `recording_task.file_path`。这种方式实现简单，但会把主机系统盘直接变成视频盘。

#### 2.2 新设计

录播改为“**本地临时落地 + 异步上传对象存储 + 成功后清理本地文件**”。

建议流程：

1. 录播过程中仍先写本地临时目录。
2. 录播完成后，由独立上传 worker 负责分片上传到 OSS/S3 兼容对象存储。
3. 上传成功后写回 `recording_task`：
   - `storage_provider`
   - `object_key`
   - `storage_bucket`
   - `storage_status`
   - `size_bytes`
   - `duration_secs`
   - `uploaded_at`
4. 本地文件进入待删除队列。
5. 删除前保留一个短暂缓冲窗口，便于失败重试。

#### 2.3 安全策略

录播文件默认不暴露公网永久地址。用户端如需播放，建议走：

- 后端鉴权
- 后端生成短期签名 URL
- 前端播放签名 URL

管理员和普通用户的下载/播放权限继续由后端控制，不能把对象存储路径直接当作前端权限边界。

### 三、Redis 负责共享缓存与实时态

#### 3.1 本阶段 Redis 的职责

Redis 只承担以下轻量职责：

1. 房间列表接口短缓存。
2. 当前直播间实时态缓存。
3. 任务调度锁与幂等保护。
4. 可选：热点排行榜短缓存。

Redis **不是** 原始事件主存储，也不是录播元数据主存储。

#### 3.2 房间列表缓存迁移

当前 `/api/rooms` 和 `/api/rooms/stats` 已有进程内 `Map` 短缓存。问题是：

- 仅当前进程可见。
- 多实例时不共享。
- 重启即失效。
- 容易让 Web 进程继续承担缓存策略。

建议迁到 Redis，键设计如下：

- `room:list:v1:{userId}:{page}:{limit}:{search}`
- `room:stats:v1:{userId}:{page}:{limit}:{search}:{sort}`

TTL 建议：

- 默认 `10 ~ 30` 秒
- 编辑房间、删除房间、停止房间、重命名房间时主动失效

#### 3.3 当前直播实时态

建议增加以下 Redis key：

- `room:live:{roomId}`
- `room:live:summary`

内容包括：

- `is_live`
- `last_event_at`
- `viewer_count`
- `current_session_id`
- `chat_count_live`
- `gift_value_live`

TTL 建议 `60 ~ 120` 秒，由采集进程持续续期。

这层数据只服务“当前直播态展示”和“低延迟列表补充”，不替代 PostgreSQL 历史真相。

### 四、重型任务独立成 worker

#### 4.1 当前问题

当前系统已有多类后台任务：

- 用户语言分析
- 会话归并
- `room_stats` 刷新
- `user_stats` 刷新
- `global_stats` 刷新
- 过期房间清理

这些任务如果继续放在 Web 进程里，会与线上请求抢 CPU、连接池和 I/O。

#### 4.2 新设计

引入独立 worker 进程，例如：

- `workers/stats_worker.js`
- `workers/recording_upload_worker.js`
- `workers/maintenance_worker.js`

职责划分：

- Web：在线接口、轻量写入、Socket 广播
- Stats Worker：定时汇总、刷新缓存表
- Upload Worker：录播上传、补写元数据、清理本地文件
- Maintenance Worker：过期清理、回填、巡检

#### 4.3 锁与并发控制

定时任务需要分布式锁，避免多实例重复跑。优先采用以下任一方式：

1. PostgreSQL advisory lock
2. Redis 锁

本阶段推荐优先用 **PostgreSQL advisory lock**，因为依赖更少，且关键任务本就围绕主库执行。

### 五、把“全量重算”改成“增量汇总 + 定时校正”

#### 5.1 目标

保留 `room_stats`、`user_stats`、`global_stats` 三张缓存表，但把它们的更新方式从“定时扫全表”改为“增量汇总 + 夜间校正”。

#### 5.2 新增派生表

建议增加以下派生层：

##### `room_minute_stats`

按房间和分钟聚合：

- `room_id`
- `stat_minute`
- `chat_count`
- `gift_value`
- `member_count`
- `like_count`
- `max_viewer_count`
- `gift_user_count`
- `chat_user_count`
- `updated_at`

作用：

- 详情页趋势图
- 最近 24 小时/7 天统计
- 全局小时/周分布的上游数据源

##### `session_summary`

按场次聚合：

- `session_id`
- `room_id`
- `start_time`
- `end_time`
- `duration_secs`
- `chat_count`
- `gift_value`
- `member_count`
- `max_viewer_count`
- `top_gifter_user_id`
- `top_gifter_value`
- `updated_at`

作用：

- 房间详情页历史场次列表
- 单场复盘
- 房间长期表现汇总

##### `user_room_gift_daily`（可选）

按用户、房间、自然日聚合：

- `stat_date`
- `user_id`
- `room_id`
- `gift_value`
- `gift_count`

作用：

- 用户画像
- Top room 计算
- `user_stats` 增量刷新

#### 5.3 更新策略

推荐策略：

1. **分钟级增量**：worker 每 1 分钟消费上一分钟事件，写 `room_minute_stats`。
2. **会话级增量**：会话结束时刷新 `session_summary`。
3. **小时级汇总**：`global_stats` 从 `room_minute_stats` 生成，而不是直接扫 `event`。
4. **夜间校正**：每天凌晨针对近 3 天窗口做一次校正重算，处理迟到事件和少量漏算。

这样可以让大多数在线接口不再碰大表。

### 六、Web 接口改造原则

#### 6.1 保持 API 契约稳定

以下接口保持现有对外结构不变：

- `/api/rooms`
- `/api/rooms/stats`
- `/api/rooms/:id/detail` 相关详情接口
- `/api/analysis/users`
- `/api/analysis/stats`

变更应该尽量发生在服务端实现层，而不是前端协议层。

#### 6.2 查询优先级

统一采用以下优先级：

1. Redis 热缓存
2. 汇总表 / 派生表
3. 最近窗口实时补算
4. 必要时才读原始 `event`

#### 6.3 在线接口禁止触发全量刷新

Web 接口可以读取缓存，但不能承担全表汇总、全量修复、全库分析等重任务。所有这类动作都应转为：

- 管理员手动触发到 worker
- 或由 worker 定时执行

## 数据生命周期

### 原始事件

建议默认保留策略：

- 活跃房间原始事件：保留 `90 天`
- 重点房间可配置延长到 `180 天`
- 失活且无人关联房间：继续保留现有 `7 天` 清理策略

### 聚合数据

建议保留更长：

- `room_minute_stats`：`180 天`
- `session_summary`：长期保留
- `room_stats / user_stats / global_stats`：长期保留

### 录播文件

建议按业务等级配置保留：

- 默认 `7 ~ 30 天`
- 重点录播手动永久保留
- 过期文件由对象存储生命周期规则 + worker 双保险清理

## 运维与观测

### 必须新增的指标

建议新增以下内部指标：

1. 每分钟事件写入量
2. `event` 分区大小与行数
3. Redis 命中率
4. 房间列表接口缓存命中率
5. Worker 执行耗时与失败次数
6. 录播上传成功率、重试次数、上传滞后时长
7. 本地临时录播目录占用
8. PostgreSQL 慢查询、分区扫描范围、索引命中情况

### 告警建议

至少增加：

- 系统盘可用空间低于 `20 GiB`
- 本地临时录播目录高于阈值
- Redis 不可用
- Worker 连续失败
- 对象存储上传堆积
- 房间列表接口 P95 超阈值

## 实施步骤

### Phase 0：基线与开关

目标：先量化，再改造。

动作：

1. 增加关键指标和慢查询采样。
2. 给新逻辑加 feature flag。
3. 补一份当前索引使用率和缓存命中率基线。

### Phase 1：录播外置

目标：先解除磁盘风险。

动作：

1. 为 `recording_task` 增加对象存储元数据字段。
2. 新增上传 worker。
3. 录播完成后异步上传到对象存储。
4. 上传成功后清理本地文件。

收益：最快，最直接，风险最低。

### Phase 2：Redis 共享缓存

目标：把短缓存从单进程 `Map` 提升为共享缓存。

动作：

1. 接入 Redis。
2. 改造房间列表和房间统计缓存。
3. 新增实时态 key。
4. 实现缓存失效策略。

### Phase 3：worker 化

目标：Web 不再承担重任务。

动作：

1. 把 `room_stats`、`user_stats`、`global_stats` 刷新迁到 worker。
2. 把会话归并、过期清理迁到 worker。
3. 用 advisory lock 控制单实例执行。

### Phase 4：`event` 模型补列 + 回填

目标：消除 JSON 依赖，统一新旧分析口径。

动作：

1. 给 `event` 增加新列。
2. 新写入逻辑直接写展开列。
3. 对历史数据做有限窗口回填。
4. 逐步把统计 SQL 从 `data_json` 切到显式列。

### Phase 5：`event` 分区化

目标：控制大表体积和扫描范围。

动作：

1. 建分区父表和新分区。
2. 双写或切换写入到分区表。
3. 迁移历史数据。
4. 重建必要索引。
5. 验证查询计划。

### Phase 6：增量汇总层

目标：把在线查询从原始大表上移开。

动作：

1. 新建 `room_minute_stats`、`session_summary` 等派生表。
2. 实现分钟级或场次级增量汇总。
3. 改写 `global_stats`、详情页趋势、用户画像等读取路径。
4. 夜间做校正重算。

## 风险与缓解

### 风险 1：对象存储上传失败导致录播状态不一致

缓解：

- 本地文件在上传成功前不删除。
- `recording_task` 增加明确状态机。
- 上传 worker 支持重试和死信记录。

### 风险 2：分区迁移期间查询行为变化

缓解：

- 先对只读查询压测。
- 保留回滚脚本。
- 在切换期对比新旧 SQL 结果。

### 风险 3：Redis 不可用导致缓存层失效

缓解：

- Redis 只做缓存，不做真相存储。
- Redis 故障时允许降级回 PostgreSQL 直查。

### 风险 4：增量汇总口径与历史全量口径不一致

缓解：

- 设计校验脚本，对抽样房间做新旧结果 diff。
- 初期保留“手动全量重算”后台入口，仅管理员可用。

## 验收标准

### 性能目标

1. `/api/rooms` 和 `/api/rooms/stats`
   - Redis 命中时 P95 < `200 ms`
   - 未命中但走汇总层时 P95 < `800 ms`

2. 房间详情页
   - 最近场次与趋势图查询 P95 < `1.5 s`

3. 统计任务
   - 不再由 Web 进程触发全量刷新
   - worker 执行期间不明显影响在线接口

4. 存储
   - 系统盘长期保留 `>= 20 GiB` 可用空间
   - 录播完成后 24 小时内完成上传和本地清理

### 正确性目标

1. 新旧统计口径抽样 diff 误差可控。
2. 会话归并和录播记录不丢失。
3. 普通用户与管理员的数据权限边界不变。

## 与 InfluxDB 的关系

本阶段不引入 InfluxDB，原因不是它不好，而是**它不适合作为当前系统的主业务库**。

当前系统的核心数据不是纯时序指标，而是：

- 用户、房间、会话、权限、支付等事务实体
- 按房间、用户、礼物、会话做条件筛选和聚合
- 需要严格后端鉴权和后台管理

这些需求更适合 PostgreSQL 主库。

如果未来要引入第二个分析库，也应该在本方案稳定后，再评估是否需要面向 OLAP 的专用分析引擎，而不是在当前阶段先换主库。

## 结论

方案A 是当前系统最合理的演进方向。

它不是“换一个数据库”就结束，而是一次**职责重分配**：

- PostgreSQL 保留主库角色
- Redis 承担共享缓存和实时态
- 对象存储接管录播文件
- Worker 接管重任务
- `event` 改造为可维护的大表体系
- 汇总层替代在线全表扫描

这条路线的优点是：

1. 与现有代码兼容。
2. 能直接解决当前最急的磁盘风险。
3. 能在不推翻系统的前提下，把容量和可维护性往前推进一个台阶。
4. 为后续是否引入更强分析引擎预留空间，但不提前引入不必要复杂度。
