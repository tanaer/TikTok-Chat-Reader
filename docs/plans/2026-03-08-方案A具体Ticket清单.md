# 方案A具体 Ticket 清单

## 说明

本文把方案A拆成可以直接进入 Jira、禅道、GitHub Projects 或飞书项目的具体 Ticket。

默认约束如下：

1. 保留 `PostgreSQL` 作为主库。
2. 不在本阶段切换到 `MySQL` 或 `InfluxDB`。
3. 优先保留现有用户态 API 契约。
4. 所有上线项都必须支持灰度、回滚和观测。
5. 任何涉及 `routes/`、支付、认证、用户数据的 Ticket，在完成前都要做一次 response-surface review。

## Ticket 约定

### 优先级

- `P0`：必须优先处理，直接影响线上风险或后续所有阶段。
- `P1`：高优先级，建议进入最近两个迭代。
- `P2`：结构性优化，建议在基础设施稳定后推进。
- `P3`：增强项，可跟随主链路实施。

### 角色建议

- `BE`：后端开发
- `DBA`：数据库/数据工程
- `OPS`：运维/平台
- `QA`：测试

### 状态建议

每个 Ticket 建议初始状态为 `Todo`，并在系统里额外维护：

- `Blocked`
- `Ready for Dev`
- `Ready for Test`
- `Ready for Release`
- `Done`

## Phase 0

### A0-01 补齐关键链路结构化指标与日志

- **角色**：BE / OPS
- **目标**：为后续灰度和回滚提供最小可观测能力。
- **主要工作**：
  - 给房间列表接口增加耗时、缓存命中、异常统计。
  - 给录播链路增加本地落盘、上传、清理阶段日志。
  - 给统计任务增加执行耗时、成功次数、失败次数日志。
- **影响文件**：`server.js`、`manager.js`、`recording_manager.js`、新增 `services/metricsService.js`
- **依赖**：无
- **验收标准**：
  - 日志里可以定位接口、录播、统计三条主链路的耗时和结果。
  - 默认开关关闭时，不改变现有业务结果。

### A0-02 增加方案A相关 Feature Flag 与环境变量

- **角色**：BE
- **目标**：让方案A所有新能力都可灰度、可回滚。
- **主要工作**：
  - 在 `.env.example` 中新增 Redis、OSS、Worker、分区、增量汇总相关变量。
  - 为新逻辑建立统一读取入口，避免散落硬编码。
- **影响文件**：`.env.example`、`server.js`、`recording_manager.js`、新增 worker 文件
- **依赖**：无
- **验收标准**：
  - 新环境变量有清晰注释。
  - 默认关闭时，系统行为与当前线上一致。

### A0-03 输出当前线上基线快照

- **角色**：OPS / DBA
- **目标**：固化改造前基线，方便后续做效果对比。
- **主要工作**：
  - 记录当前 `event` 体量、索引体量、磁盘可用空间。
  - 记录 `/api/rooms`、`/api/rooms/stats` 的 P95。
  - 记录统计任务耗时和录播目录大小。
- **影响文件**：新增 `docs/plans/2026-03-08-方案A上线前基线.md` 或内部表格
- **依赖**：A0-01
- **验收标准**：
  - 能拿出改造前可追溯的基线文档。

## Phase 1

### A1-01 扩展 `recording_task` 存储元数据字段

- **角色**：DBA / BE
- **目标**：为对象存储接入和本地清理提供元数据基础。
- **主要工作**：
  - 在 `recording_task` 新增对象存储、上传状态、清理状态相关字段。
  - 保留 `file_path` 兼容字段。
- **影响文件**：`db.js`
- **依赖**：A0-02
- **验收标准**：
  - 新字段迁移可重复执行。
  - 旧数据不受影响。

### A1-02 实现对象存储服务层

- **角色**：BE / OPS
- **目标**：统一 OSS/S3 兼容对象存储的上传、删除、签名逻辑。
- **主要工作**：
  - 新增 `services/objectStorageService.js`。
  - 新增 `services/recordingStorageService.js` 封装录播对象命名、上传和签名地址逻辑。
- **影响文件**：新增 `services/objectStorageService.js`、新增 `services/recordingStorageService.js`
- **依赖**：A0-02
- **验收标准**：
  - 可在测试环境上传、删除、生成短期签名地址。
  - 错误场景有明确错误码和日志。

### A1-03 新增录播上传 Worker 骨架

- **角色**：BE
- **目标**：将录播上传从 Web/录播进程中剥离。
- **主要工作**：
  - 新增 `workers/recording_upload_worker.js`。
  - 轮询待上传记录，执行上传并回写状态。
- **影响文件**：新增 `workers/recording_upload_worker.js`、`db.js`
- **依赖**：A1-01、A1-02
- **验收标准**：
  - Worker 能识别待上传任务。
  - 上传成功/失败均可正确回写状态。

### A1-04 重构录播状态机并接入上传流程

- **角色**：BE
- **目标**：让录播生命周期清晰可追踪。
- **主要工作**：
  - 将 `recording_task` 状态从单一 `recording/completed/failed` 扩展为本地完成、上传中、已上传、清理待处理等状态。
  - 在录播结束时将任务推进到上传流程。
- **影响文件**：`recording_manager.js`、`db.js`、`workers/recording_upload_worker.js`
- **依赖**：A1-03
- **验收标准**：
  - 一条录播任务的全链路状态变化清晰可见。
  - 失败重试不会污染正常任务。

### A1-05 增加录播播放/下载签名地址接口

- **角色**：BE
- **目标**：用户通过后端鉴权访问对象存储中的录播文件。
- **主要工作**：
  - 增加面向用户/管理员的录播访问接口。
  - 普通用户只返回短期签名地址，不暴露原始对象路径。
  - 完成一次 response-surface review。
- **影响文件**：`routes/user.js`、`routes/admin.js`、可选新增 `routes/recordings.js`、`services/recordingStorageService.js`
- **依赖**：A1-04
- **验收标准**：
  - 普通用户只能访问其有权限的录播。
  - 不返回内部 bucket、原始凭证或调试信息。

### A1-06 增加本地录播清理与生命周期策略

- **角色**：BE / OPS
- **目标**：确保本地目录只保留短期临时文件。
- **主要工作**：
  - 上传成功后延迟清理本地文件。
  - 增加失败重试窗口和兜底清理任务。
  - 结合对象存储生命周期做长期保留策略。
- **影响文件**：`workers/recording_upload_worker.js`、新增 `workers/maintenance_worker.js`、`recording_manager.js`
- **依赖**：A1-04
- **验收标准**：
  - 本地录播目录占用开始稳定回落。
  - 上传成功后不会长期残留大文件。

## Phase 2

### A2-01 接入 Redis 客户端与缓存封装

- **角色**：BE / OPS
- **目标**：提供统一的 Redis 访问能力。
- **主要工作**：
  - 新增 `services/redisClient.js`。
  - 新增 `services/cacheService.js`，封装 JSON、TTL、删除、批量失效。

- **依赖**：A0-02
- **验收标准**：
  - 可在 Redis 可用和不可用两种情况下稳定运行。
  - 封装层对业务代码暴露简单接口。

### A2-02 将房间列表缓存迁移到 Redis

- **角色**：BE
- **目标**：把 `Map` 缓存升级为跨进程共享缓存。
- **主要工作**：
  - 替换 `server.js` 中的 `roomListResponseCache`。
  - 保持当前 key 维度和 TTL 语义。
- **影响文件**：`server.js`、`services/cacheService.js`
- **依赖**：A2-01
- **验收标准**：
  - 多实例下命中逻辑一致。
  - Redis 故障时能降级到数据库直查。

### A2-03 新增直播实时态缓存服务

- **角色**：BE
- **目标**：将当前直播间实时状态从单机内存抽离。
- **主要工作**：
  - 新增 `services/liveStateService.js`。
  - 在采集链路写入 `is_live`、`last_event_at`、`viewer_count`、`gift_value_live` 等实时态。
- **影响文件**：新增 `services/liveStateService.js`、`auto_recorder.js`、`recording_manager.js`
- **依赖**：A2-01
- **验收标准**：
  - `/api/rooms`、`/api/rooms/stats` 可读取 Redis 实时态。
  - 不再只依赖单机 `activeConnections` 衍生结果。

### A2-04 整理缓存失效与降级策略

- **角色**：BE
- **目标**：保证缓存正确性和可维护性。
- **主要工作**：
  - 对房间新增、删除、编辑、停用、别名修改、录播状态变化等动作统一失效。
  - 明确 Redis 不可用时的降级行为。
- **影响文件**：`server.js`、`routes/rooms.js`、`routes/user.js`、可选 `routes/admin.js`
- **依赖**：A2-02、A2-03
- **验收标准**：
  - 关键写操作后缓存不会出现长时间脏数据。
  - 降级场景下接口仍正确返回。

### A3-03 新增维护 Worker 并迁出清理/归并任务

## 上线顺序建议

- **先上线**：Phase 0 + Phase 1 + Phase 2
- **第二批上线**：Phase 3 + Phase 4
- **最后上线**：Phase 5 + Phase 6

原因：

1. 第一批直接缓解磁盘风险和接口缓存压力。
2. 第二批解决 Web 进程负担和事件模型不一致。
3. 第三批属于结构性演进，收益大，但迁移和验证成本最高。

## 结论

如果你要把这份清单直接发给研发团队建单，建议先建三个 Epic：

1. `EPIC-A1A2A3`：录播外置、Redis、Worker 化
2. `EPIC-A4A5`：事件模型与分区演进
3. `EPIC-A6`：增量汇总与读路径改造

其中，**A1-01 到 A3-05** 是最值得优先立项的一组 Ticket。
