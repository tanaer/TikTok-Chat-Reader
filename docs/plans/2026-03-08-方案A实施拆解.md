# 方案A实施拆解与落地清单

## 说明

本文是 `docs/plans/2026-03-08-方案A架构优化-spec.md` 的实施拆解版。

目标不是重复 Spec，而是把方案A拆成可以直接排期、建任务、指派开发和验收的工作包。

本文默认遵循以下原则：

1. 先处理磁盘和线上风险，再处理结构性优化。
2. 先拆职责，再做大表演进。
3. 优先保留现有 API 契约。
4. 所有改动都以可灰度、可回滚、可观测为前提。

## 总体拆解

方案A拆成 6 个阶段，建议顺序如下：

1. **Phase 0：基线、开关、观测补齐**
2. **Phase 1：录播外置到对象存储**
3. **Phase 2：Redis 共享缓存与实时态**
4. **Phase 3：重任务 worker 化**
5. **Phase 4：`event` 模型补列与统计去 JSON 化**
6. **Phase 5：`event` 分区化**
7. **Phase 6：增量汇总层落地**

其中：

- **最优先上线**：Phase 1、Phase 2、Phase 3
- **结构性收益最大**：Phase 5、Phase 6
- **贯穿式前置条件**：Phase 0

## 建议排期

按中等强度投入估算：

- Phase 0：`1 ~ 2` 天
- Phase 1：`3 ~ 5` 天
- Phase 2：`2 ~ 4` 天
- Phase 3：`3 ~ 5` 天
- Phase 4：`3 ~ 6` 天
- Phase 5：`5 ~ 8` 天
- Phase 6：`5 ~ 10` 天

总工期大致为 `3 ~ 6` 周，取决于：

- 是否要做历史数据回填
- 是否要在本阶段一次性完成分区迁移
- 是否要同步补齐后台管理和运维页面

## 工作流依赖图

依赖关系建议如下：

- Phase 0 是全部阶段的前置条件。
- Phase 1 可独立启动，优先级最高。
- Phase 2 与 Phase 3 可以并行推进，但最好先完成 Redis 接入，再做 worker 锁和状态协作。
- Phase 4 应在 Phase 6 之前完成，避免汇总层继续依赖 JSON 字段。
- Phase 5 最好在 Phase 4 之后做，避免分区后再大规模补列与回填。
- Phase 6 建议在 Phase 3、4、5 具备基础后落地。

建议落点：

#### P0-2 Feature Flag 与环境变量整理

建议新增环境变量：

影响文件：

#### P0-3 基线文档输出

内容：

- 记录当前 `event` 体量、磁盘占用、索引占用、房间列表 P95、统计任务耗时。

交付：

- 可单独落一份运维基线文档，或者先记录到内部表格。

### 验收标准

- 新开关默认关闭时，线上行为不变。
- 关键链路能看见结构化日志和基线数据。

## Phase 1：录播外置到对象存储

### 目标

先解除本地系统盘压力，把录播文件从“长期落地”改成“临时落地 + 异步上传 + 清理”。

### 任务包

#### P1-1 扩展 `recording_task` 表结构

内容：

新增字段建议：

- `storage_provider`
- `storage_bucket`
- `object_key`
- `storage_status`
- `storage_error`
- `uploaded_at`
- `delete_local_after_upload`
- `local_cleanup_at`
- `duration_secs`
- `mime_type`

影响文件：

- `db.js`

说明：

- 原有 `file_path` 先保留，作为兼容字段。
- 新逻辑允许 `file_path` 指向本地临时路径，`object_key` 指向永久对象路径。

#### P1-2 新增对象存储服务层

内容：

新增建议文件：

- `services/objectStorageService.js`
- `services/recordingStorageService.js`

职责：

- 上传文件
- 查询对象元数据
- 生成签名下载链接
- 删除对象

#### P1-3 新增录播上传 worker

内容：

新增建议文件：

- `workers/recording_upload_worker.js`

职责：

- 轮询 `recording_task` 中待上传记录
- 上传成功后回写元数据
- 标记失败并重试
- 在缓冲时间后清理本地文件

#### P1-4 录播结束状态机调整

内容：

把录播状态细化为：

- `recording`
- `completed_local`
- `uploading`
- `uploaded`
- `upload_failed`
- `cleanup_pending`
- `archived`

影响文件：

- `recording_manager.js`
- `db.js`
- 新 worker 文件

#### P1-5 播放/下载鉴权接口梳理

内容：

- 普通用户访问录播时由后端生成签名 URL。
- 管理员保留更完整的查看能力。
- 不直接向普通用户暴露原始对象路径。

影响文件：

- `routes/admin.js`
- `routes/user.js`
- 或新增 `routes/recordings.js`

### 依赖

- 依赖 Phase 0 的配置开关与日志能力。

### 验收标准

- 录播完成后可自动上传对象存储。
- 上传成功后本地文件在规定窗口内自动清理。
- 本地磁盘曲线不再随着录播线性增长。
- 普通用户仍只能访问其有权限的录播内容。

## Phase 2：Redis 共享缓存与实时态

### 目标

把当前进程内 `Map` 缓存迁成共享缓存，并为实时房间态提供统一读写入口。

### 任务包

#### P2-1 接入 Redis 基础设施

新增建议文件：

- `services/cacheService.js`

职责：

- JSON 读写包装
- TTL 封装

#### P2-2 房间列表缓存迁移

当前代码锚点：

- `server.js:235`
- `server.js:248`

内容：

- 继续支持统一失效。

影响文件：

#### P2-3 当前直播实时态缓存

内容：

新增建议文件：

- `services/liveStateService.js`

写入来源：

- `auto_recorder.js`
- `recording_manager.js`

读取来源：

- `/api/rooms`
- `/api/rooms/stats`

#### P2-4 缓存失效规范

内容：

统一整理哪些动作会触发失效：

- 房间新增
- 房间删除
- 房间编辑
- 房间停用
- 别名修改
- 录播状态变化

影响文件：

- `server.js`
- `routes/rooms.js`

### 验收标准

- Redis 不可用时接口仍可回退到 PostgreSQL 直查。
- 当前直播态不再依赖单机内存保存。

## Phase 3：重任务 worker 化

### 目标

把统计刷新、归并、清理等任务从 Web 进程中剥离。

### 任务包

#### P3-1 任务清单与归属调整

当前代码锚点：

- `server.js:3242`
- `server.js:3247`
- `server.js:3321`
- `server.js:3331`

迁出任务：

- 用户语言分析
- 会话归并
- `room_stats` 刷新
- `user_stats` 刷新
- `global_stats` 刷新

#### P3-2 新建 worker 入口

新增建议文件：

- `workers/stats_worker.js`
- `workers/maintenance_worker.js`
- `bin/start_stats_worker.js`
- `bin/start_maintenance_worker.js`

#### P3-3 抽离任务调度与执行器

内容：

把当前散落在 `server.js` 的 `setInterval` 和 `setTimeout` 抽到任务执行器中。

建议新增文件：

- `services/jobRunner.js`
- `services/jobLockService.js`

#### P3-4 并发控制与互斥锁

内容：

- 优先采用 PostgreSQL advisory lock。
- 每类任务有独立 lock key。
- 同一时间只允许一个实例执行全局任务。

#### P3-5 管理员触发入口改造

内容：

- 保留管理员手动刷新入口。
- 手动触发改为“发起任务”，不在 Web 请求内同步跑完整流程。

影响文件：

- `server.js`
- `routes/admin.js`
- `manager.js`

### 依赖

- 建议依赖 Phase 2 完成，这样 Redis 和锁服务都可复用。

### 验收标准

- Web 进程不再定时执行重任务。
- worker 宕机不会拖垮在线接口。
- 手动刷新改为异步任务后，接口响应明显更快。

## Phase 5：`event` 分区化

### 目标

把当前单大表改造成可维护、可清理、可控索引范围的分区表。

### 任务包

#### P5-1 分区 DDL 设计

新增建议文件：

- `scripts/migrate_event_partitioning.js`
- `docs/plans/事件分区DDL草案.md`（可选）

内容：

- 设计 `event` 父表
- 建按月分区
- 设计分区索引模板
- 建未来分区自动创建脚本

#### P5-2 新写入切换

内容：

- 让新写入落到分区父表。
- 应用层尽量不感知具体子分区。

影响文件：

- `db.js`
- `manager.js`

#### P5-3 历史数据迁移

内容：

- 分批迁移历史 `event` 数据到分区结构。
- 控制单批次大小，避免长事务。
- 迁移期间保留回滚策略。

#### P5-4 索引重建与瘦身

内容：

- 针对分区重建最小必要索引。
- 删除低命中、高体积、重复语义索引。
- 给冷热分区采用不同策略。

#### P5-5 查询计划验收

内容：

- 对 `/api/rooms/stats`
- `/api/rooms`
- 房间详情页
- 用户画像
- `global_stats` 刷新链路

逐条看 `EXPLAIN ANALYZE`，确认发生分区裁剪。

### 依赖

- 建议在 Phase 4 后实施。

### 验收标准

- 新写入稳定落到分区表。
- 近 7 天和近 30 天查询只命中相关分区。
- 冷数据清理和迁移不再依赖操作整张大表。

## Phase 6：增量汇总层

### 目标

把在线读路径从“原始大表即时报表”切换到“派生汇总 + 最近窗口补算”。

### 任务包

#### P6-1 建立派生表

建议新增表：

- `room_minute_stats`
- `session_summary`
- `user_room_gift_daily`（可选）

影响文件：

- `db.js`

#### P6-2 增量汇总 worker

内容：

- 每分钟消费上一分钟窗口的 `event`
- 刷新 `room_minute_stats`
- 场次结束时刷新 `session_summary`
- 夜间做短窗口校正

建议落点：

- `workers/stats_worker.js`
- `manager.js`
- 或新增 `services/statsAggregationService.js`

#### P6-3 改写读路径

优先改造路径：

- 房间详情趋势图
- `global_stats`
- 房间长期表现
- 用户画像相关榜单

原则：

- 优先读派生表
- 必要时补最近窗口
- 避免直接扫全部 `event`

#### P6-4 校验与回归

内容：

- 抽样比对新旧统计结果
- 允许保留管理员手动全量校正入口
- 统计差异超阈值要报警

### 依赖

- 依赖 Phase 3、Phase 4、Phase 5 提供基础设施。

### 验收标准

- 大多数在线统计接口不再直接读取全量 `event`。
- 统计结果与历史口径差异在可接受范围内。
- `global_stats` 构建耗时显著下降。

## 任务粒度建议

如果按 Jira / 禅道 / GitHub Project 建任务，建议每个 Phase 再拆成如下粒度：

- 一类任务只改一块主能力
- 每个任务最多跨 `3 ~ 5` 个主要文件
- DDL 任务与业务逻辑任务分开
- worker 基础设施与具体统计逻辑分开
- 灰度开关和回滚方案单独成任务

建议不要把下面内容混成一个任务：

- Redis 接入 + worker 化
- `event` 补列 + 分区迁移
- 录播上传 + 播放鉴权 + 生命周期清理

## 推荐的实际执行顺序

如果目标是**最快降低线上风险**，建议顺序是：

1. Phase 0
2. Phase 1
3. Phase 2
4. Phase 3
5. Phase 4
6. Phase 5
7. Phase 6

如果目标是**最快提升接口性能**，建议顺序是：

1. Phase 0
2. Phase 2
3. Phase 3
4. Phase 4
5. Phase 6

但结合本系统的现实情况，**仍推荐把 Phase 1 提前**，因为录播文件对磁盘的压力比数据库增长更急。

## 角色分工建议

建议最少按三类角色推进：

1. **后端主程**
   - Redis 接入
   - worker 化
   - 统计读路径改造
   - API 保持兼容

2. **数据/DB 工程**
   - `event` 补列
   - 回填脚本
   - 分区 DDL
   - 索引瘦身

3. **运维/平台**
   - 对象存储接入
   - worker 部署
   - Redis 部署
   - 告警与观测

如果团队人少，也建议按这个逻辑切任务，而不是按文件夹切任务。

## 每阶段完成后的检查点

### Checkpoint A：Phase 1 完成后

应确认：

- 本地录播目录开始回落
- 上传失败有可见告警
- 普通用户权限未越界

### Checkpoint B：Phase 3 完成后

应确认：

- Web 进程 CPU 与 I/O 抖动下降
- 重任务不再阻塞接口
- 管理后台手动触发仍可用

### Checkpoint C：Phase 5 完成后

应确认：

- 最近窗口查询发生分区裁剪
- 索引大小增速下降
- 冷数据维护成本下降

### Checkpoint D：Phase 6 完成后

应确认：

- 主要在线分析接口已切到汇总层
- 统计准确率稳定
- `event` 只在少数补算路径中被直接读取

## 结论

方案A的实施重点不是“同时做很多事”，而是按顺序拆掉几个最重的耦合点：

1. **本地磁盘和录播强耦合**
2. **Web 进程和重任务强耦合**
3. **在线接口和全量事件扫描强耦合**
4. **统计口径和 JSON 字段强耦合**
5. **大表与单表索引体积强耦合**

只要这五个耦合点拆开，系统就会从“单机越跑越重”变成“可以分阶段扩容和演进”的状态。
