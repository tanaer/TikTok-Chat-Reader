# 方案A开发总计划

## 目标

本计划用于指导 `tiktok-monitor` 的方案A实施。目标是在不更换主库品类的前提下，优先解决磁盘风险、在线接口与重任务耦合、事件大表维护成本和统计链路扩展性问题。

本计划是当前开发阶段的唯一主计划文件。后续新 session 应优先读取本文，再结合以下文档推进：

- `docs/plans/2026-03-08-方案A架构优化-spec.md`
- `docs/plans/2026-03-08-方案A实施拆解.md`
- `docs/plans/2026-03-08-方案A具体Ticket清单.md`

## 当前判断

### 现状

- 主库继续使用 `PostgreSQL`。
- 当前最紧急问题不是 CPU，而是本地磁盘和录播文件占用。
- `event` 已是系统最大热点表，后续必须走“补列、去 JSON、分区、汇总层”路线。
- Web 进程仍承担定时重任务，需要迁到独立 worker。
- 房间列表缓存仍是单进程 `Map`，后续应迁到 Redis。

### 总体策略

按“先止血，再拆耦合，再做结构演进”的顺序推进：

1. 先补观测、开关和基线。
2. 先把录播迁到对象存储，释放磁盘压力。
3. 再把缓存和实时态迁到 Redis。
4. 再把重任务迁到 worker。
5. 然后补齐 `event` 模型并逐步去 JSON。
6. 最后推进 `event` 分区和增量汇总层。

## 开发阶段与顺序

### Phase 0：基线、开关、观测补齐

目标：为后续所有改造提供可灰度、可回滚、可对比的基础。

本阶段完成项：

- 增加结构化日志和关键耗时统计。
- 增加方案A相关 Feature Flag。
- 固化当前线上基线快照。

### Phase 1：录播外置到对象存储

目标：尽快把录播从本地系统盘迁出。

本阶段完成项：

- 扩展 `recording_task` 元数据字段。
- 新增对象存储服务层。
- 新增录播上传 worker。
- 调整录播状态机。
- 完成签名地址接口和本地清理策略。

### Phase 2：Redis 共享缓存与实时态

目标：把当前单进程缓存提升为共享缓存，并抽离直播实时态。

本阶段完成项：

- 接入 Redis 客户端和缓存封装。
- 迁移 `/api/rooms` 与 `/api/rooms/stats` 缓存。
- 新增直播实时态缓存服务。
- 梳理缓存失效与降级策略。

### Phase 3：重任务 worker 化

目标：Web 进程只负责在线请求，重任务交给 worker。

本阶段完成项：

- 抽离任务调度与锁服务。
- 新增统计 worker。
- 新增维护 worker。
- 管理员手动刷新改为异步任务。

### Phase 4：`event` 模型补列与统计去 JSON 化

目标：统一新旧事件模型，减少统计对 `data_json` 的依赖。

本阶段完成项：

- 为 `event` 增加分析用展开列。
- 改造 `logEvent(...)` 写入链路。
- 编写历史事件补列脚本。
- 改写核心统计 SQL。

### Phase 5：`event` 分区化

目标：控制大表扫描范围和索引体积。

本阶段完成项：

- 落地按月分区 DDL。
- 切换新写入到分区父表。
- 迁移历史数据。
- 执行索引瘦身。
- 建立查询计划回归用例。

### Phase 6：增量汇总层落地

目标：让在线统计读路径从原始大表上移开。

本阶段完成项：

- 新增 `room_minute_stats` 与 `session_summary`。
- 建立增量汇总服务层。
- 在统计 worker 中接入汇总任务。
- 将 `global_stats` 和详情趋势切到汇总层。
- 增加新旧口径校验与管理员校正入口。

## 当前推荐开工顺序

新 session 应优先按以下顺序推进：

1. `A0-01` 补齐关键链路结构化指标与日志
2. `A0-02` 增加方案A相关 Feature Flag 与环境变量
3. `A1-01` 扩展 `recording_task` 存储元数据字段
4. `A1-02` 实现对象存储服务层
5. `A1-03` 新增录播上传 Worker 骨架

原因：

- 这组任务不依赖大规模读路径重构。
- 这组任务能最快降低本地磁盘风险。
- 这组任务能为后续 Redis、worker 和统计层改造打基础。

## 当前状态

截至本计划创建时：

- Spec 已完成。
- 实施拆解已完成。
- 具体 Ticket 清单已完成。
- 代码尚未开始进入方案A第一批实施。

因此，新 session 的默认任务是：**从 Phase 0 与 Phase 1 的首批 Ticket 开始实现，而不是继续补文档。**

## 开工约束

新 session 开工时应遵循以下约束：

1. 先读计划与 handoff，再动代码。
2. 每次只落一个小批次任务，避免跨越太多 Phase。
3. 涉及 `routes/`、用户数据、支付、认证的改动，必须做 response-surface review。
4. 先做最小实现，再补验证和文档。
5. 对象存储、Redis、worker 相关能力必须有 feature flag。
